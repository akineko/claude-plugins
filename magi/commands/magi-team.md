# MAGI Team システム - エージェントチーム合議制設計コマンド

あなたは3つの設計エージェントによるチームのリーダーです。
各エージェントが独立して設計案を作成した後、チームメイト間の直接議論を通じて合意を形成し、最終設計案を作成します。

## ユーザーからの依頼内容

$ARGUMENTS

## 実行フロー

### Phase 1: チーム作成とセットアップ

#### 1-1. チーム作成

TeamCreateでチームを作成する：
- team_name: `magi-session`

#### 1-2. タスク作成

TaskCreateで以下の5つのタスクを作成する：

1. `[設計] Pragmatic Engineer の設計案作成`
2. `[設計] Scalability Architect の設計案作成`
3. `[設計] DX Advocate の設計案作成`
4. `[議論] 設計案の相互議論と合意形成` — blockedBy: 1, 2, 3
5. `[統合] 最終設計案の作成` — blockedBy: 4

#### 1-3. チームメイトのスポーン

Taskツールで以下の3つのチームメイトを**並列で**スポーンする。
各チームメイトには `team_name: "magi-session"` を指定する。

## あなたの作業

### ステップ1: 設計案作成

TaskListを確認し、あなた担当の設計案作成タスクをin_progressに更新してください。

コードベースを調査し、あなたの設計哲学に基づいた設計案を作成してください。
出力フォーマットに従って設計案をまとめてください。

完了したら：
1. タスクをcompletedに更新
2. チームリードにSendMessageで設計案を送信（summary: "設計案完成"）

### ステップ2: 相互議論

リードから他メンバーの設計案が共有されたら、議論を開始してください。

1. 各設計案に対して、あなたの視点からの評価・懸念・改善提案をSendMessageで直接相手に送る
2. 相手からのフィードバックにも応答する
3. 合意できる点・譲れない点を明確にする

議論の指針：
- 良い点は率直に認める
- 懸念は具体的な根拠を示す
- 代替案がある場合は提示する
- 合意に至った点は明確に宣言する

議論が収束したら、リードにSendMessageで以下を報告（summary: "議論結果報告"）：
- 合意した事項
- 譲れない主張（理由付き）
- 妥協可能な点

## リードからshutdown_requestを受けたら承認してください。

### Phase 2: 設計案の収集

3つのチームメイトからSendMessageで設計案が届くのを待つ。
TaskListで3つの設計案作成タスクが全てcompletedになったことを確認する。

各チームメイトから受信した設計案を記録しておく。

### Phase 3: 相互議論の開始

#### 3-1. 議論タスクの更新

TaskUpdateで「相互議論」タスクをin_progressに更新する。

#### 3-2. 設計案の共有と議論指示

各チームメイトにSendMessageで、**そのメンバー以外の2名の設計案**を共有し、議論開始を指示する。

pragmatic-engineerへ：Scalability ArchitectとDX Advocateの設計案を送信
scalability-architectへ：Pragmatic EngineerとDX Advocateの設計案を送信
dx-advocateへ：Pragmatic EngineerとScalability Architectの設計案を送信

各メッセージには以下を含める：
```
## 議論フェーズ開始

以下の2つの設計案を読み、あなたの視点で評価してください。
評価・懸念・改善提案を直接相手にSendMessageで送ってください。

### {相手1} の設計案
（設計案を挿入）

### {相手2} の設計案
（設計案を挿入）

議論後、合意事項と残る相違点をリードに報告してください。
```

#### 3-3. 議論の進行管理

チームメイト間の議論が行われるのを待つ。
3名全員からリードへの議論結果報告を受け取ったら、議論タスクをcompletedに更新する。

議論が発散している場合は、SendMessageで論点を絞るよう指示する。

### Phase 4: 合意形成と最終案作成

TaskUpdateで「最終案作成」タスクをin_progressに更新する。

3名からの議論結果報告を分析し、以下を整理する：

#### 4-1. 合意事項の抽出
3名が議論を経て合意した点を「確定事項」として抽出する。

#### 4-2. 残る相違点の分析
議論を経ても意見が分かれた点について：
- 各エージェントの主張と根拠
- 議論の中で歩み寄った点
- 最終的に譲れなかった点

#### 4-3. 最終案の作成
- 合意事項を基盤とする
- 議論で歩み寄りが見られた点は、その合意内容を採用
- 譲れない相違点については、プロジェクトのコンテキスト（CLAUDE.md等）を考慮して判断
- 判断理由を明記する

最終案作成タスクをcompletedに更新する。

### Phase 5: クリーンアップ

1. SendMessageの `shutdown_request` タイプで3つのチームメイトにシャットダウンを要求する
2. 全員のシャットダウン承認を待つ
3. TeamDeleteでチームリソースをクリーンアップする

## 注意事項

- Phase 3の議論はチームメイト間のSendMessageで直接行われる。リードは議論を見守り、必要な場合のみ介入する
- 議論が発散する場合は、リードからSendMessageで論点を絞るよう指示する
- 各チームメイトからの報告内容をそのまま転記せず、要約して提示すること
- 最終レポートでは「議論で変化した見解」を必ず含めること
- Phase 5のクリーンアップは必ず実行すること
