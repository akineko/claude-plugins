# 影響分析書の作成

要求仕様書を入力として、コードベースを調査し、設計に必要な事実情報を集約した影響分析書を作成する。

## 目的

設計判断に必要な情報（影響範囲、設計制約、依存関係）を事実として集約する。
この文書は実装設計書の入力として使用される。

**このスキルが出力するもの**: 事実、制約、依存関係
**このスキルが出力しないもの**: 設計案、選択肢の比較、推奨アプローチ

## ワークフロー概要

```
入力（要求仕様書パス）
  │
  ├─ Phase 1: 初期化・変更一覧の作成     ← メインエージェント
  │
  ├─ Phase 2: Research（コード調査）      ← codebase-investigator
  │     └─ 変更ごとに影響ファイルを調査
  │
  ├─ Phase 3: 分析・文書作成              ← メインエージェント
  │     └─ 設計制約・依存関係・分割推奨
  │
  ├─ Phase 4: ユーザー確認               ← ループ
  │
  成果物: .claude/tmp/{dir}/impact-analysis.md
```

## 制約

- 設計判断を含めない。「AとBの選択肢がある」ではなく「Xという事実がある」と記述する
- Research結果を変更ごとに整理する。フラットなファイル一覧にしない
- 推奨設計単位は依存関係分析の事実に基づく。好みや慣例に基づかない

### 禁止表現

以下の表現は設計判断の示唆であり、影響分析書に含めてはならない:

- 「〜が候補」「〜が考えられる」 → 選択肢の提示は設計の責務
- 「〜が合理的」「〜が望ましい」 → 価値判断は設計の責務
- 「〜すべき」「〜を推奨する」 → 方針の提案は設計の責務
- 具体的な新規ファイル名・コンポーネント名 → 命名は設計の責務。「新規ディレクトリが必要」は事実だが「XxxContainer.tsxを作成」は設計

## Phase 1: 初期化・変更一覧の作成

### 1.1 入力の取得

- `$ARGUMENTS` がファイルパス → 読み込む
- `$ARGUMENTS` が空 → AskUserQuestion で入力を求める

### 1.2 成果物パスの決定

要求仕様書と同じディレクトリに `impact-analysis.md` を配置する。

例: 要求仕様書が `.claude/tmp/feature-x/request-spec.md` → `.claude/tmp/feature-x/impact-analysis.md`

### 1.3 変更一覧の作成

要求仕様書の要件を以下の種別に分類する:

- **追加**: 新しい機能・エンドポイント・UIコンポーネント
- **変更**: 既存機能の修正・拡張
- **削除**: フィールド・機能の廃止

分類は要件レベルで行う。実装手段（「中間テーブルを追加」等）には言及しない。
各変更にIDを振る（C1, C2, ...）。このIDは以降のフェーズで参照に使う。

**例（良い）**: `[追加] 商品をカテゴリ一覧に表示`
**例（悪い）**: `[追加] ProductListItems.tsxコンポーネントを作成` ← 実装手段

### 1.4 Research範囲の判定

変更一覧から影響するサービス/レイヤーを推定する:
- 2つ以上のサービスに跨る → 並列Research
- 単一サービス内 → 単一Research

## Phase 2: Research（コード調査）

`references/research-prompt.md` を読み込み、変数を埋めて `codebase-investigator` に依頼する。

### 単一Researchの場合

```
Agent(
  subagent_type="codebase-investigator",
  prompt=research-prompt.md の変数を埋めたもの
)
```

### 並列Researchの場合

サービス/レイヤーごとにエージェントを並列起動する。

```
Agent(name="research-backend", subagent_type="codebase-investigator", prompt=...)
Agent(name="research-frontend", subagent_type="codebase-investigator", prompt=...)
```

Research結果を変更IDごとに整理して保持する（Phase 3で使用）。

## Phase 3: 分析・文書作成

`assets/template.md` をテンプレートとして読み込み、以下の手順で影響分析書を作成する。

### 3.1 セクション1: 変更一覧

Phase 1で作成した変更一覧をテンプレートの形式で記載する。

### 3.2 セクション2: 影響範囲

Research結果を変更ごとに整理する。各変更について:

- **変更対象ファイル**: ファイルパス、現在の役割、必要な変更を記載
- **参考にすべき既存パターン**: 類似実装のファイルパスとパターン概要
- **注意点**: その変更に固有の技術的注意点

同じファイルが複数の変更に関係する場合、それぞれの変更セクションに記載する（後のセクション4で依存関係として分析する根拠になる）。

**「必要な変更」列の記載ルール**:
- 既存ファイルに対しては変更の性質と対象を書く（例: 「商品用データ取得の追加」「legacyFlagフィールドの削除」）
- 新規ファイル・ディレクトリの場合は目的のみ書く（例: 「注文関連のAPIフック・コンポーネント群」）
- 具体的なファイル名・コンポーネント名・関数名は書かない。命名は設計の責務

### 3.3 セクション3: 設計制約

Research結果と要求仕様書から、設計時に考慮すべき制約を抽出する。

| 制約の種類 | 含めるべき内容 | 含めてはいけない内容 |
|-----------|-------------|-------------------|
| 技術的制約 | フレームワークの制約、既存の命名規則、DBの型制約 | 「Xを使うべき」という推奨 |
| ビジネス制約 | 要求仕様書の優先度、必須/任意の区別 | 実装順序の提案 |
| 互換性制約 | 既存APIの利用者、データマイグレーションの必要性 | マイグレーション方法の選択 |
| 横断的な注意点 | 複数変更に共通する事実 | 設計方針 |

### 3.4 セクション4: 依存関係分析

変更間の依存関係を分析する。

**依存グラフ**: 変更間の依存をASCII図で示す。以下の凡例を必ず図の前に記載する:

```
凡例: A → B は「AはBに依存する（Bが先に完了している必要がある）」
```

- 全ての依存関係をグラフに含める。テキストで言及した依存がグラフに欠けていてはならない
- 依存がない変更ペアは線で結ばない
- 独立したグループは空行で分離する

**独立性の評価**: 以下の基準で各変更ペアの独立性を評価する。

| 基準 | 独立と判断する条件 |
|------|-----------------|
| データモデルの独立性 | 変更対象が別テーブル・別カラムで、相互に参照しない |
| デプロイの独立性 | 一方を先にリリースしても他方に影響しない |
| レイヤー依存の方向 | 一方のAPI設計が確定しないと他方の設計が不安定になる関係がない |

### 3.5 セクション5: 推奨設計単位

依存関係分析に基づき、独立して設計可能な単位を提案する。

各単位には以下を記載:
- 含む変更（IDで参照）
- 対象レイヤー
- スコープ概要（1-2文）

設計単位が1つだけの場合（全変更が密結合している場合）、その旨を記載して分割しない。

設計単位間に順序依存がある場合、依存関係分析の事実に基づいて実施順序を記載する。
順序の根拠は依存関係の事実（「AのAPIレスポンス型がBの入力に必要」等）のみ。効率性・合理性の判断は含めない。

### 3.6 ファイル保存

完成した影響分析書を成果物パスに保存する。

## Phase 4: ユーザー確認

### 4.1 提示

影響分析書の要点を簡潔に提示する:

- 変更一覧（セクション1）の概要
- 推奨設計単位（セクション5）の分割案
- 特に注意が必要な設計制約があれば強調

提示後、`AskUserQuestion` で確認を求める:

```
影響分析書を確認してください: {成果物パス}

修正したい箇所があれば指摘してください。
問題なければ「OK」と回答してください。
```

### 4.2 フィードバック対応

| ユーザーの回答 | アクション |
|---|---|
| 「OK」「問題なし」「LGTM」等 | Phase 5 へ |
| 変更一覧の修正 | 修正を反映し、影響範囲・依存関係にも波及する場合は追加Researchを検討 |
| 分割案の修正 | 依存関係分析を再評価し、修正を反映 |
| 影響範囲の追加・削除 | 修正を反映 |

修正後は 4.1 に戻る。

## Phase 5: 完了

1. 成果物のファイルパスを案内する
2. 推奨設計単位の概要を1-3行で提示する
3. 次ステップへの案内: 「この分析書を元に、各設計単位ごとに実装設計書を作成できます」

## エラーハンドリング

### Researchが失敗した場合
- エラー内容をユーザーに報告する
- 調査範囲を絞って再試行するか、手動で情報を補完するか確認する

### 変更一覧の分類が困難な場合
- 要求仕様書の記述が曖昧な場合、AskUserQuestion で確認する
- 1つの要件が複数の変更種別にまたがる場合、分割して記載する

### 中断からの再開
- 成果物ファイルが既に存在する場合、ユーザーに上書きするか確認する
