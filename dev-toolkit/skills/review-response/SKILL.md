# Review Response

レビューコメントを受け取り、各コメントについて客観的に対応の必要性を判断し、対応が必要な場合は改善案を提案する。

レビュアーの指摘やコメントに含まれる改善案をそのまま受け入れるのではなく、コードベースの実態・設計意図・トレードオフを踏まえて独立した設計判断を行うことがこのスキルの核心。

## ワークフロー

### Step 1: レビューコメントの取得

#### GitHub PR の場合

PR番号またはURLが指定された場合、`gh` コマンドでコメントを取得する。

```bash
# PRのレビューコメント取得
gh api repos/{owner}/{repo}/pulls/{number}/comments --jq '.[] | {id: .id, path: .path, line: .original_line, body: .body, user: .user.login}'

# PR全体へのコメント（issue comments）も取得
gh api repos/{owner}/{repo}/issues/{number}/comments --jq '.[] | {id: .id, body: .body, user: .user.login}'
```

bot による自動コメント（lint、CI結果等）は除外する。判断基準は `.user.type == "Bot"` またはコメント内容から明らかに自動生成と判別できるもの。

#### テキスト貼り付けの場合

ユーザーが会話中に貼り付けたコメントをそのまま使う。複数コメントがある場合は番号を振って管理する。

### Step 2: 対象コードの把握

レビューコメントが言及しているファイル・箇所を特定し、以下を確認する：

1. **指摘箇所のコード** — 該当ファイルの該当行とその周辺
2. **関連コード** — import先、呼び出し元、型定義、テストなど、指摘の妥当性を判断するために必要な文脈
3. **設計の経緯** — git log/blame で変更の意図や背景が読み取れる場合は参考にする

ここで重要なのは、レビュアーが見ていない文脈まで含めてコードを理解すること。レビュアーは差分を中心に見ているため、既存コードの制約や設計意図を見落としていることがある。

### Step 3: 各コメントの評価

コメントごとに以下の観点で評価する。

#### 3-1. コメントの分類

まずコメントの性質を把握する：

| 種別 | 説明 |
|------|------|
| **指摘** | バグ、設計上の問題、改善すべき点の報告 |
| **質問** | コードの意図や設計判断の確認 |
| **提案** | 具体的な改善案を伴う指摘 |
| **感想/FYI** | 対応不要の情報共有やコメント |

#### 3-2. 客観的な対応判断

各コメントについて、コードの実態に基づいて判断する。レビュアーの指摘が的確なこともあれば、文脈不足で的外れなこともある。以下の基準で判定する：

| 判定 | 基準 |
|------|------|
| **対応する** | 指摘が妥当であり、コードを改善すべき明確な理由がある |
| **要議論** | 指摘の観点は理解できるが、トレードオフがあり一概に判断できない |
| **対応しない** | 指摘が事実誤認、既に対処済み、過剰な最適化、設計意図と相反するなど |
| **回答のみ** | 質問や感想への応答で済むもの |

判定において意識すべきこと：

- **提案をそのまま採用しない**: レビュアーの改善案が含まれていても、それが最善かは別問題。提案の意図（何を解決したいのか）を汲み取った上で、より適切な方法がないか検討する
- **権威に流されない**: 誰が言ったかではなく、何を言っているかで判断する
- **既存設計との整合性**: 局所的には正しくても、コードベース全体の設計方針と矛盾する変更は慎重に判断する
- **変更の影響範囲**: 指摘への対応が他の箇所に波及する場合、その影響も考慮する

#### 3-3. 改善案の検討（対応する/要議論の場合）

対応すると判断した場合、具体的な改善案を検討する：

1. レビュアーの提案がある場合、その案を評価する（採用/改良/別案）
2. コードベースの既存パターンや設計方針に沿った解決策を検討する
3. 改善案が複数ある場合は比較し、推奨を示す

改善案はコードレベルで具体的に示す。「こうした方がいい」ではなく、実際のコード変更をイメージできる粒度で記述する。

### Step 4: レポートの出力

```markdown
# Review Response

## 概要
- **対象**: {PRの情報 or テキスト入力}
- **コメント数**: {N}件
- **判定内訳**: 対応 {n} / 要議論 {n} / 対応しない {n} / 回答のみ {n}

## 対応する

### コメント #{番号}: {コメントの要約}
- **ファイル**: `{file}:{line}`（該当する場合）
- **レビュアーのコメント**: {原文の要約}
- **判断理由**: {なぜ対応すべきか}
- **改善案**: {具体的な改善内容}
  - レビュアーの提案と異なる場合、差異とその理由を記載

## 要議論

### コメント #{番号}: {コメントの要約}
- **ファイル**: `{file}:{line}`（該当する場合）
- **レビュアーのコメント**: {原文の要約}
- **論点**: {何がトレードオフなのか}
- **選択肢**:
  - A: {案Aとそのメリット/デメリット}
  - B: {案Bとそのメリット/デメリット}
- **推奨**: {どちらが良いと考えるか、その理由}

## 対応しない

### コメント #{番号}: {コメントの要約}
- **レビュアーのコメント**: {原文の要約}
- **判断理由**: {対応しない理由を具体的に}
- **返信案**: {レビュアーへの返信として使える説明文}

## 回答のみ

### コメント #{番号}: {コメントの要約}
- **レビュアーのコメント**: {原文の要約}
- **回答案**: {質問や感想への返信}
```

#### 出力時の注意

- 該当するコメントがないカテゴリはセクションごと省略する
- 「対応しない」の判断理由は特に丁寧に書く — ユーザーがレビュアーに説明する材料になる
- 改善案のコードは、ユーザーが「これを適用して」と言えばそのまま実装に移れる具体性を持たせる

## 禁止事項

- コードの直接修正 — 判断と提案のみ行い、修正はユーザーの承認後に別途実施する
- レビュアーへの直接返信 — 返信案を提示するのみ

## 使用例

```
/review-response 123                        # PR #123 のレビューコメントを評価
/review-response https://github.com/.../pull/123  # URL指定
/review-response                            # テキストで貼り付けたコメントを評価
```
