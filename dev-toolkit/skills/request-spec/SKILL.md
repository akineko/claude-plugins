# 要求仕様書を作成する

PMから伝えられた要件を構造化し、設計フェーズへの引き継ぎ文書を作成する。

## 重要な制約

- **技術的判断は行わない**: 実装方法、アーキテクチャ等の設計はこのフェーズでは決めない
- **PMからの技術指定は記録する**: 「このAPIを使って」等の指定があれば要件として残す
- **推測は明示する**: 推測で埋めた箇所は「（推測）」とマークし、確認時に伝える
- **TBDは有効な回答**: 決まっていないことは無理に埋めず「TBD」とする

## 使用方法

```
/request-spec [実現したいことの説明]
```

引数がない場合は入力を求める。

## 作成フロー

### 1. 入力の受け取り

- 引数があればそれを使用
- なければ「実現したいこと」を入力するよう求める

### 2. 種別の判定

入力内容から種別を判定：
- **機能追加**: 新しい機能を追加
- **機能修正**: 既存機能の変更・改善
- **バグ修正**: 不具合の修正

判定が難しい場合は AskUserQuestion で確認。

### 3. 出力先の作成

- `assets/template.md` をテンプレートとして使用
- `.claude/tmp/<機能名>/request-spec.md` を作成
- `<機能名>` は英数字とハイフン（例: `user-authentication`）

### 4. セクション毎の作成と確認

各セクションで以下を繰り返す：

1. 入力情報を元にセクション内容を作成してファイルに追記
   - 情報がある箇所: そのまま記載
   - 推測で埋めた箇所: 「（推測）」マーク付き
   - 情報がない箇所: 「TBD」
2. 追加内容をユーザーに提示
   - 推測箇所は明示して確認を促す
3. ユーザーの確認を待つ
   - OK → 次のセクションへ
   - 修正要望 → 修正して再提示

**セクション順序:**

1. 基本情報
2. 背景・目的
3. 要求内容（種別に応じた形式）
4. 制約条件
5. 未決定事項
6. 参考情報

### 5. 完了前チェック

全セクション完了後、以下を確認：
- 技術的な実装詳細（設計判断）を書いていないか
- 推測箇所に「（推測）」マークがあるか
- 背景・目的が「なぜ」に答えているか

### 6. 完了

ファイルパスを案内し、次のステップ（設計フェーズ）への引き継ぎを促す。

## 使用例

### 例1：引数付きで起動

ユーザーの発言：`/request-spec ユーザーがダッシュボードから3Dモデルをアップロードできるようにしたい`
実行内容：
1. 種別を「機能追加」と判定
2. `.claude/tmp/3d-model-upload/request-spec.md` を作成
3. セクション毎にユーザーと対話しながら仕様書を完成させる

### 例2：引数なしで起動

ユーザーの発言：`/request-spec`
実行内容：
1. 「実現したいこと」の入力を求める
2. 入力に基づき種別を判定し、以降は例1と同様

## トラブルシューティング

### 種別の判定が困難

原因：入力が抽象的で、機能追加・修正・バグ修正のいずれか判断できない
解決策：AskUserQuestion で「これは新機能の追加ですか？既存機能の修正ですか？バグの修正ですか？」と確認する

### ユーザーが技術的な実装方法を指定してきた場合

原因：PMが技術的な指定を含めて要件を伝えている
解決策：技術指定はそのまま「技術的な指定」セクションに記録し、設計判断は行わない
