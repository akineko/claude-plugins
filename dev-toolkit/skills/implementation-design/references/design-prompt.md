## 入力の使い分け

- **影響分析書が提供されている場合**: 影響分析書のセクション2（影響範囲）をコード調査結果として使用する。セクション3（設計制約）を設計判断の制約として考慮する。設計対象の単位に含まれる変更IDのみをスコープとする
- **関連コード調査結果が提供されている場合**: 従来通り調査結果を使用する
- **要求仕様**: どちらのパスでも背景・目的の理解に使用する

## 先行設計書との整合性

先行設計書が提供されている場合、以下のルールに従う:

- 先行設計書で確定した設計判断（セクション2）に矛盾する判断を行わない
- 先行設計書で定義されたデータ構造（セクション3の型定義・DBスキーマ・API仕様）はそのまま使用する。同じフィールドやスキーマに対して異なる定義を出力しない
- 先行設計書のAPI仕様（レスポンス形状・エンドポイントURL）に依存する場合、先行設計書の定義を前提として設計する
- 先行設計書のスコープに含まれる変更を重複して設計しない。自設計単位のスコープ外の変更は先行設計書に委ねる

## 出力フォーマット

以下のセクション構成で出力せよ。セクション1（ゴール）は確定済みのため出力に含めない。セクション2から開始する。セクションの追加・省略・順序変更は禁止。

### 2. 設計判断

要求仕様を実現するための技術的選択を列挙する。
このセクションが設計書の核。実装者がコードを書く途中で判断に迷ったとき、ここに立ち返る。

各判断を以下の形式で記載する:

```
### [判断事項を短く表現]

**選択**: [選んだ選択肢]
**根拠**: [なぜこれが最善か。複数行になってよい]
**却下案**: [検討して却下した案] → [却下した理由]
```

**ルール**:
- 要求仕様に既に書かれている内容を設計判断として記載するな。「要件Xをどう実現するか」の技術的選択が設計判断であり、「要件Xを実現する」は設計判断ではない。例: 要件が「一覧画面に商品を表示する」なら、それ自体は判断ではない。「統合リストのデータ構造をdiscriminated unionにする」は判断
- 「〜が必要」は根拠ではない。それは要件の言い換えにすぎず、判断の指針にならない。「AはBという性質があり、Cという制約下ではDが最善」が根拠
- 却下案がない判断は判断ではない。選択肢が1つしかないなら書く必要はない

### 3. データ構造定義

型定義・DBスキーマ・API仕様の**目標状態**を記載する。

#### コード記載量の基準

| 変更の種類 | 記載する内容 |
|-----------|-------------|
| **新規ファイル** | 完全なコード |
| **既存ファイルへの追加** | 追加部分のみ。配置場所が分かる最小限のコンテキスト（周辺の既存コード1-2行）を付ける |
| **既存フィールドの削除のみ** | 「モデルXからフィールドYを削除」と記述すれば十分。完全なモデル定義は不要 |
| **DBマイグレーション（データ移行あり）** | 完全なSQLを記載する（不可逆操作のため省略不可） |

#### 3.3 API仕様

**`METHOD /path`** — [エンドポイントの目的]

```typescript

// レスポンス
```

**ルール**:
- **変更前のコード（Before）は書くな。** AIはReadツールで現在のコードを読める
- 新規定義と既存定義の変更を明確に区別する
- 該当なしのサブセクションは「該当なし」と書く

### 4. モジュール設計

新規作成・大幅変更するモジュールの責務・処理フロー・依存関係を記載する。

#### [モジュール名]

- **責務**: [1文]
- **処理フロー**: [番号付きリスト、3-5ステップ]
- **依存**: [モジュール名のリスト]

**ルール**:

- 既存パターンを踏襲する場合、参考ファイルのパスを1つ示せ。パターンの説明は不要（AIが読める）
- セクション3のコード変更で自明な軽微変更（プロパティ追加のみ等）はここに書くな

### 5. データフロー

主要なユースケースのデータフローをASCII図で示す。
ユースケースごとに1つの図。補足説明は図の前後に最小限。

```
[入力]
  │
  ↓
[処理]
  │
  ├─ [分岐A]
  └─ [分岐B]
```

### 6. リスクと対策

実装上のリスクとそれに対する具体的な対策を記載する。

#### [リスク名]

**内容**: [何が起こりうるか]
**対策**: [具体的な行動。「テストを十分に行う」は対策ではない]

**ルール**:
- 発生確率が低く影響も軽微なリスクは書くな
- 対策は実行可能な具体的行動のみ

## 禁止事項

以下に違反した出力はやり直しになる。context window の無駄遣いを防ぐために厳守せよ。

1. **ファイル一覧セクションの禁止**: 「編集対象ファイル」「参照ファイル」「影響ファイル」のようなファイルパスだけを列挙するセクションを作るな。ファイルパスはセクション3-4の文脈内で必要な箇所にのみ記載する
2. **既存コードの転記禁止**: 変更前のコードや既存ファイルの全体をコピーするな。追加・変更部分のみ記載せよ（「コード記載量の基準」テーブル参照）
3. **情報量ゼロの記述の禁止**: 「エラー処理: 既存パターンに委譲」「依存: prisma」のような、読んでも判断材料にならない記述は書くな
4. **導入文の禁止**: 「このセクションでは〜を説明します」「以下に〜を示します」のような前置きは書くな

## 品質基準

出力完了前に自己検証せよ:

- セクション2（設計判断）に要求仕様の言い直しが混入していないか？各判断は技術的選択か？
- セクション3のコードは追加・変更部分のみか？既存ファイルの内容を丸ごとコピーしていないか？
- ファイルパスだけを列挙したテーブルが存在しないか？
- 各セクションが前のセクションと情報を重複していないか？
