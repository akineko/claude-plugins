# Task Development

実装設計書のフェーズ、または直接の要件に基づき、専門サブエージェントに委譲して開発を進める。
依存関係のないタスクは最大限並列化し、効率的に実装を完了させる。

## サブエージェント一覧

| エージェント | 役割 | 複数同時起動 |
|-------------|------|:----------:|
| `td-task-designer` | 設計書なしの場合に要件整理・実装計画を策定 | - |
| `td-frontend-developer` | TDDでフロントエンド実装 | 可 |
| `td-backend-developer` | TDDでバックエンド実装 | 可 |
| `td-test-runner` | 全体テスト実行・結果報告 | - |
| `td-requirements-reviewer` | 元の要件/設計書と実装結果の突き合わせ | - |

## ワークフロー判定

ユーザーの入力から以下を判定する：

| 入力パターン | パス |
|-------------|------|
| 実装設計書パス + フェーズ指定あり | → パターンA |
| 要件の直接記述（設計書なし） | → パターンB |
| 判断できない | → ユーザーに確認 |

---

## パターンA: 実装設計書あり

### Step 1: フェーズ解析とタスク分割

1. 指定された実装設計書を読み、該当フェーズのタスクリストを抽出する
2. 各タスクをFE/BEに分類する（設計書に記載があればそれに従う）
3. タスク間の依存関係を分析し、並列実行グループを構成する

#### 並列実行グループの構成

タスク間の依存関係を分析し、依存のないタスクを同一グループにまとめる。
ただし、同じファイルを編集する可能性があるタスク同士は同一グループに入れない（並列実行時の競合を防ぐため）。
判断基準: 共有の型定義、API契約ファイル、設定ファイル、マイグレーションなど、複数タスクから参照・編集されうるファイルがあれば別グループにする。

```
例: フェーズ内タスク A(FE), B(BE), C(BE), D(FE)
    依存: D → B（DはBの完了が必要）、A/B/Cは独立
    共有ファイル: AとCが同じ型定義ファイルを編集する

    グループ1: [A, B]    ← 2並列（Cは共有ファイル競合のため分離）
    グループ2: [C]       ← グループ1完了後
    グループ3: [D]       ← グループ2完了後（Bへの依存）
```

分析結果に基づき直接実行に進む。

### Step 2: 共通実行フローへ

タスク分割が完了したら「共通実行フロー」の Phase 1 から実行する。

---

## パターンB: 設計書なし（小規模タスク）

### Step 1: 実装計画の策定

`td-task-designer` を起動し、要件の整理と実装計画を策定する。

#### 指示

````
以下の要件を整理し、実装計画を策定してください。

## 要件
{ユーザーが伝えた要件をそのまま記載}
````

### Step 2: 計画の確認

task-designer の策定した実装計画をユーザーに提示し、承認を得る。

- 計画をそのまま表示し「この計画で実装を進めます。修正があれば指摘してください」と伝える
- ユーザーが修正を求めた場合、task-designer に再依頼する
- ユーザーが承認したら Step 3 に進む

### Step 3: 共通実行フローへ

task-designer の計画を設計書と同等に扱い、「共通実行フロー」の Phase 1 から実行する。

---

## 共通実行フロー

パターンA/Bのタスク分割完了後、以下のフローを順に実行する。

### Phase 1: 開発（グループ順に実行）

各グループ内のタスクを並列でサブエージェントに委譲する。

#### エージェントへの指示書の構成

各エージェントに渡すプロンプトには以下を含める：

````
以下のタスクをTDDで実装してください。

## 実装設計書
{設計書パスを指定し、エージェント自身に読ませる。パターンBの場合は task-designer の計画内容}

## 実装対象
{該当タスクの内容。設計書内のセクション参照で指定}

## インタフェース定義
{設計書に記載されたインタフェース・データモデルのうち、このタスクに関連する部分}

## 依存タスクの実装結果
{先行グループで完了したタスクのうち、このタスクが依存するものの概要。なければ省略}
````

#### 起動ルール

- 同一グループ内のエージェントは **1回のレスポンスで全て同時に起動する**
- FE タスクには `td-frontend-developer`、BE タスクには `td-backend-developer` を使う
- 同じ種別のエージェントを複数同時に起動してよい（例: backend-developer x 2）
- グループ内の全エージェント完了後、次のグループに進む

### Phase 2: 全体テスト

全グループの開発完了後、`td-test-runner` を起動し全体テストを実行する。

#### テスト実行の指示

````
以下のプロジェクトの全体テストを実行してください。

## 対象
{プロジェクトルートパス}

## 今回の変更概要
{実装したタスクの概要リスト}
````

#### テスト失敗時

1. 失敗内容を分析し、原因となるタスク（FE/BE）を特定する
2. 該当の developer エージェントに修正を依頼する（失敗内容と関連コードを指示に含める）
3. 修正完了後、再度 `td-test-runner` で全体テストを実行する
4. 全テスト通過まで繰り返す（ユーザー確認は不要）。ただし **修正→再テストは最大3回** まで。3回で解消しない場合はそれまでの失敗経緯をユーザーに報告し、判断を仰ぐ

### Phase 3: 要件レビュー

全テスト通過後、`td-requirements-reviewer` を起動する。

#### レビュー指示

````
以下の実装が要件を満たしているかレビューしてください。

## 実装設計書
{設計書パス、またはパターンBの場合は task-designer の計画内容}

## 実装対象フェーズ
{フェーズ名/番号、またはパターンBの場合は要件の要約}

## 実装されたファイル
{変更されたファイルの一覧。git diff --name-only 等で取得}
````

### Phase 4: 結果報告

以下の形式でユーザーに報告する：

```markdown
# 開発完了レポート

## 実装概要
- **設計書**: {パス、またはパターンBの場合は「task-designer による計画」}
- **対象フェーズ**: {フェーズ名、またはパターンBの場合は要件の要約}
- **実装タスク数**: {N}

## 実装したタスク
| # | タスク | 種別 | 実行グループ |
|---|--------|------|:----------:|
| 1 | {タスク名} | FE/BE | 1 |
| ... | ... | ... | ... |

## テスト結果
- **全体テスト**: Pass / {失敗→修正の経緯があれば記載}

## 要件レビュー結果
{requirements-reviewer の結果をそのまま提示}
```

---

## 禁止事項

- スキル本体が直接コードを書くこと → 必ず developer エージェントに委譲する
- developer エージェントの逐次起動 → 同一グループ内は必ず同一レスポンスで全て起動する
- テスト失敗時のユーザー確認 → 自動で差し戻し・修正・再テストする
- コード品質・セキュリティのレビュー → 別スキル（code-review等）の責務
