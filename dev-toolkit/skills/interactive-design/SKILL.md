# 対話形式の実装設計書作成

影響分析書の変更一覧（C1, C2...）を1項目ずつ対話で設計を詰め、最終的に既存の実装設計書と同じフォーマットの設計書を生成する。

影響範囲が広い・設計の選択肢が多いタスクに向いている。影響範囲が明確で一括設計できるタスクには `implementation-design` や `multi-perspective-design` を使う。

## ワークフロー概要

```
入力（影響分析書パス）
  │
  ├─ Phase 1: 初期化
  │     ├─ 影響分析書 + 要求仕様書を読み込み
  │     ├─ 変更項目の選択（複数選択可）
  │     └─ 依存関係チェック + 警告
  │
  ├─ Phase 2: 項目別設計（選択項目ごとにループ）
  │     ├─ 2.1 設計提案（推奨案 + 別案）
  │     ├─ 2.2 ユーザー確認（OK / 修正指示）
  │     └─ 2.3 確定 → 次の項目へ（確定済み設計を引き継ぐ）
  │
  ├─ Phase 3: 統合（確定済み設計 → セクション1-6の設計書）
  │     ├─ 横断的整合性チェック
  │     └─ ユーザー確認
  │
  ├─ Phase 4: Planning（セクション7: 実装フェーズ）
  │
  成果物: {影響分析書と同じディレクトリ}/implementation-design-C{ids}.md
```

## 制約

- コードベース調査が必要な場合のみ `codebase-investigator` を使用する。影響分析書の情報で足りる場合は不要
- サブエージェントはファイル書き込み禁止。内容生成のみ担当する
- メインエージェントがファイルの作成・保存を行う
- Planning エージェントには `references/planning-prompt.md` のプロンプトを使用する

## Phase 1: 初期化

### 1.1 入力の読み込み

1. `$ARGUMENTS` から影響分析書パスを取得する。空の場合は「影響分析書のパスを引数に指定してください（例: `/interactive-design path/to/impact-analysis.md`）」と案内して終了する
2. 影響分析書を読み込む
3. 影響分析書の冒頭から `対象要求仕様書` のパスを取得し、要求仕様書を読み込む

### 1.2 変更項目の選択

影響分析書のセクション1（変更一覧）から項目リストを抽出し、AskUserQuestion（`multiSelect: true`）で設計対象を確認する。

AskUserQuestion のオプション数は2-4個の制約があるため、項目数に応じて対応を分ける:

#### 4項目以下の場合

各項目をオプションとして直接提示する:

```
AskUserQuestion(
  question: "設計対象の変更項目を選んでください",
  multiSelect: true,
  options: [
    { label: "C1: {変更内容の要約}", description: "{変更内容}" },
    { label: "C2: {変更内容の要約}", description: "{変更内容}" },
    ...
  ]
)
```

#### 5項目以上の場合

質問テキストに全項目リストを含め、選択方法をオプションとして提示する:

```
AskUserQuestion(
  question: "設計対象の変更項目を選んでください:\n\nC1: {変更内容}\nC2: {変更内容}\n...\nC{n}: {変更内容}",
  multiSelect: false,
  options: [
    { label: "全項目", description: "全ての変更項目を設計対象にする" },
    { label: "選択する", description: "対象の番号を入力（例: 1,3,5）" }
  ]
)
```

「選択する」が選ばれた場合、ユーザーが Other 経由で番号を入力する。

### 1.3 依存関係チェック

影響分析書のセクション4（依存関係分析）を使い、選択された項目の依存先が以下の状態でないか確認する:

1. **選択されていない項目に依存している**: 選択に含まれず、かつ同ディレクトリに設計済みファイル（`implementation-design-C*.md`）も存在しない
2. **設計済みファイルに含まれている**: 同ディレクトリの既存 `implementation-design-C*.md` に含まれている → 先行設計書として読み込む（セクション1-3のみ）

状態1に該当する場合、警告して確認する:

```
⚠ 選択された項目の依存関係に注意があります:

- C3 は C1 に依存していますが、C1 は選択されておらず、設計済みファイルも見つかりません

このまま進めますか？ 依存先を追加で選択することもできます。
```

### 1.4 成果物パスの決定

影響分析書と同じディレクトリに配置する。

- ファイル名: `implementation-design-C{ids}.md`
  - 選択された項目の ID を `+` で結合する（例: `implementation-design-C1+C3.md`）
  - ID は数値の昇順で並べる

### 1.5 処理順序の決定

選択された項目を依存関係に基づいて処理順序を決定する。依存先を先に処理する。依存関係がない項目間は影響分析書の ID 順に従う。

## Phase 2: 項目別設計（ループ）

選択された各項目について、以下の 2.1〜2.3 を繰り返す。

### 2.1 設計提案

項目ごとに、以下の情報を元に設計を検討し提案する。

**入力情報:**
- 影響分析書のセクション2（影響範囲）から該当項目の情報
- 影響分析書のセクション3（設計制約）
- 要求仕様書の該当箇所
- 先行項目の確定済み設計（Phase 2 で既に確定した項目 + Phase 1.3 で検出した先行設計書）

**追加調査:**
影響分析書の情報だけでは設計判断に必要な情報が不足する場合（既存パターンの詳細、型定義の構造、API の現状等）、`codebase-investigator` で追加調査を行う。

**提案フォーマット:**

```markdown
## C{n}: {変更内容}

### 推奨案

**方針**: {アプローチの概要を1-2文で}

**設計判断**:
- **{判断事項}**: {選択} — {根拠}
  - 却下案: {代替案} → {却下理由}

**データ構造の変更**: {型・スキーマ・API の変更があれば概要を記載}

**モジュール設計**: {新規・大幅変更するモジュールがあれば概要を記載}

### 別案（ある場合のみ）

**方針**: {別アプローチの概要}
**推奨案との比較**: {推奨案を推す理由、別案の利点・欠点}

### 確認ポイント（ある場合のみ）

- {ユーザー判断が必要な点}
```

提案は簡潔に。設計判断に必要な情報を過不足なく伝えることが目的で、最終的な設計書のフォーマットとは異なる。

### 2.2 ユーザー確認

提案後、ユーザーに確認する:

```
この方針で進めてよいですか？ 修正点や懸念があれば指摘してください。
```

| ユーザーの回答 | アクション |
|---|---|
| OK / LGTM / 問題なし | 2.3 へ |
| 別案を採用 | 別案ベースで推奨案を再構成し、再提案 |
| 修正指示あり | 指摘を反映して再提案 |
| 追加の質問・議論 | 回答・議論した上で必要なら再提案 |

修正指示で追加のコードベース調査が必要になった場合は `codebase-investigator` で調査してから再提案する。

### 2.3 確定

ユーザーの OK が出たら、その項目の設計を確定済みとして保持する。

次の項目に進む際、確定済みの設計判断・データ構造定義を前提として引き継ぐ。先行項目で決めた型やAPIが後続項目に影響する場合は、自動的に反映する。

全項目が完了したら Phase 3 に進む。

## Phase 3: 統合

### 3.1 設計書の生成

Phase 2 で確定した全項目の設計を、実装設計書のセクション1-6に統合する。

`references/output-format.md` を読み込み、各セクションの形式に従って設計書を生成する。

Phase 2 で確定した設計を以下の方針で統合する:
- 項目横断で重複する設計判断はまとめる
- データ構造定義は項目間で整合させて統合する
- セクション5（データフロー）は統合後の全体像として新たに作成する

### 3.2 横断的整合性チェック

統合時に以下を確認する:

- 項目間で矛盾する設計判断がないか
- データ構造定義の重複・不整合がないか（同じ型を異なる項目で別定義していないか）
- 先行項目で仮決めした内容が後続項目で変わっていないか

不整合を検出した場合、ユーザーに報告して解決方針を相談する。

### 3.3 ユーザー確認

設計書（セクション1-6）をファイルに保存し、主要ポイントを提示して確認を求める:

```
設計書を生成しました: {成果物パス}

設計判断の概要:
- {判断1のタイトル}: {選択内容}
- {判断2のタイトル}: {選択内容}
...

確認してください。修正があればセクション番号と内容を指定してください。
問題なければ「OK」で実装フェーズの分割に進みます。
```

| ユーザーの回答 | アクション |
|---|---|
| OK | Phase 4 へ |
| 修正指示あり | 該当セクションを修正し、再度確認を求める |

## Phase 4: Planning（実装フェーズ分割）

`references/planning-prompt.md` を読み込み、設計書全体（セクション1-6）を入力として `implementation-planner` に依頼する。

```
Agent(
  subagent_type="implementation-planner",
  prompt=planning-prompt.md の {設計書の内容} を埋めたもの
)
```

Planning エージェントの出力（セクション7: 実装フェーズ）を設計書の末尾に追記する。

## Phase 5: 完了

1. 成果物のファイルパスを案内する
2. ゴールと主要な設計判断の概要を3-5行で提示する
3. 次ステップへの案内: 「内容を確認して、問題なければ実装に進みましょう」

## エラーハンドリング

### コードベース調査が失敗した場合
- エラー内容をユーザーに報告する
- 調査範囲を絞って再試行するか、手動で情報を補完するか確認する

### 中断からの再開
- 成果物ファイルが既に存在する場合、ユーザーに上書きするか確認する
