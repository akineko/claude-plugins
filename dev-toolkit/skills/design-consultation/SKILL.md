ユーザーの設計相談を greenfield-architect サブエージェントに委託する。委託前に「曖昧さの解消」と「既存実装バイアスの除去」を行う。

## 絶対ルール: コード探索の禁止

このスキルの全フェーズにおいて、プロジェクトのコードや構造を探索するツール（Read, Glob, Grep, Agent 等）の使用は**禁止**。ユーザーが「このファイルを読んで」と明示的に指示した場合のみ例外とする。

理由: あなたがプロジェクトを探索した時点で、既存実装のバイアスが不可避的に混入する。greenfield-architect の価値は既存実装を知らない状態で設計することにある。あなたが探索すればその価値は失われる。

## Phase 1: 要件の明確化

ユーザーの相談内容を分析し、以下の観点で不足情報を**ユーザーへの質問のみで**補完する。

確認の観点（最低限「相談内容・ゴール」は必ず明確にすること）：

- 相談内容・達成したいゴール
- 対象スコープ
- 重要な制約条件（性能要件、セキュリティ要件、チーム規模等）

情報の取得手段は**ユーザーへの質問だけ**。コードベースの探索、ファイル構造の確認、既存実装の参照は一切行わない。「確認のために見てみよう」「参考になるかもしれない」という判断は全て禁止。ユーザーが自発的に提供した情報だけで次のフェーズに進む。

質問は一度に最大3つまでに絞り、設計判断への影響が大きい不明点から確認する。十分な情報が揃ったら Phase 2 に進む。

## Phase 2: 依頼文の構築

要件を greenfield-architect への依頼文に変換する。

### 禁止: あなた自身の判断による情報の追加

依頼文に含めて良いのは**ユーザーが自分の言葉で提供した情報のみ**。あなたの判断による情報追加は**例外なく禁止**。既存実装に引きずられた情報が混入し、理想解が得られなくなるため。

- ✅ ユーザーが「この機能のリファクタリング案が欲しい」と言って示したファイルパス → そのまま含める
- ✅ ユーザーが「これを参考に」と指定したコードや資料 → そのまま含める
- ❌ あなたが「このファイルも参考になるはず」と判断したプロジェクト内のファイル・コード
- ❌ あなたが補足として付け加えるプロジェクトの技術スタック・設計パターン・慣習
- ❌ あなたが推測した成果物のファイル名・ディレクトリ構成・命名規則
- ❌ あなたが既存コードから推測した「期待するアウトプット」の具体的な形式
- ❌ あなたが相談内容から分解した「検討すべき技術的トピック」のリスト

成果物の構成（ファイル名、ディレクトリ構造、モジュール分割等）は**設計そのもの**であり、greenfield-architect が決定すべき事項。あなたが事前に決めた時点で設計の余地を奪う。

### 必須: ユーザー提供情報の忠実な伝達

ユーザーが提供した情報は、ファイルパス・コード・技術名を含め**そのまま依頼文に含める**。ただし、ユーザーの相談意図を明確にするために表現を整理することは行う：

- ユーザー「`src/auth/` 以下のリファクタリング案が欲しい」→ 対象パスと要件をそのまま伝える
- ユーザー「Express.js の代わりに何が良いか」→ Phase 1 で要件（性能、規模等）を確認した上で、ユーザーの言葉と確認した要件を伝える

### 依頼文の構成

```
## 設計相談

### 相談内容
（要件ベースで記述）

### 制約・要件
（ユーザーから得た非実装的な制約）

### 期待するアウトプット
（ユーザーが求める回答の形式・粒度を記述する。例: 「技術選定の比較表と推奨案」「アーキテクチャの全体像」「移行戦略のステップ」等）
（⚠️ 設計判断の結論を先取りするリストを書かない。「Xの選定」「Yの実現方式」のような項目列挙は設計スコープの限定であり禁止）
（⚠️ 具体的なファイル名・パス・構成は書かない。それは設計の一部であり architect が決める）

### 参考資料（ユーザー提供）
（ユーザーが明示的に提供したコード・ファイル・資料。なければ省略）
```

## Phase 3: 依頼文の確認

構築した依頼文をユーザーに提示し、承認を得る。

- 依頼文をそのまま表示し「この内容で greenfield-architect に依頼します。修正があれば指摘してください」と伝える
- ユーザーが修正を求めた場合、依頼文を修正して再提示する
- ユーザーが承認したら Phase 4 に進む

## Phase 4: 委託と結果提示

1. Agent ツールで greenfield-architect に委託する
   - prompt: ユーザーが承認した依頼文をそのまま設定する。依頼文にない情報の付与は禁止
2. 結果をそのままユーザーに提示する（自分の意見を加えない）
3. ユーザーが追加質問や修正を求めた場合、Phase 1 に戻る
