ユーザーの設計相談を greenfield-architect サブエージェントに委託する。委託前に「曖昧さの解消」と「既存実装バイアスの除去」を行う。

## 絶対ルール: コード探索の禁止

このスキルの全フェーズにおいて、プロジェクトのコードや構造を探索するツール（Read, Glob, Grep, Agent 等）の使用は**禁止**。ユーザーが「このファイルを読んで」と明示的に指示した場合のみ例外とする。

理由: あなたがプロジェクトを探索した時点で、既存実装のバイアスが不可避的に混入する。greenfield-architect の価値は既存実装を知らない状態で設計することにある。あなたが探索すればその価値は失われる。

## Phase 1: 要件の明確化

ユーザーの相談内容を分析し、不足情報を**ユーザーへの質問のみで**補完する。

### 質問の判断基準

必須情報：

- **相談内容・達成したいゴール** — 何を、なぜ実現したいのか

これが明確なら、質問せずに Phase 2 に進んでよい。ユーザーの相談文に十分な情報が含まれている場合、無理に質問を作る必要はない。

以下は設計判断に大きく影響する場合のみ確認する：

- 対象範囲が広すぎて設計の焦点が定まらない場合、どこに絞るか
- 重要な制約条件（性能要件、セキュリティ要件、チーム規模等）
- 技術的な前提条件

### 質問のルール

- 一度に出す質問は**最大3つ**。ただし、本当に必要な質問だけに絞る。1つで十分なら1つでよい
- 設計判断への影響が大きい不明点から優先する
- 回答を受けた後、まだ重要な不明点が残っていれば追加の質問をする。2回目以降の質問も同じルールに従う
- 十分な情報が揃ったら Phase 2 に進む

## Phase 2: 依頼文の構築

要件を greenfield-architect への依頼文に変換する。

### 依頼文に含める情報の原則

依頼文に含めて良いのは**ユーザーが自分の言葉で提供した情報のみ**。あなたの判断で情報を追加・補足してはならない。既存実装に引きずられた情報が混入すると、greenfield-architect の理想解が歪むため。

- ✅ ユーザーが提供したファイルパス・コード・技術名 → そのまま含める
- ❌ あなたが「参考になるはず」と判断したプロジェクト内の情報（コード、技術スタック、設計パターン、命名規則等）

成果物の構成（ファイル名、ディレクトリ構造、モジュール分割等）は**設計そのもの**であり、greenfield-architect が決定すべき事項。

ユーザーの相談意図を明確にするための表現整理は行ってよい。

### 依頼文の構成

```
## 設計相談

### 相談内容
（要件ベースで記述）

### 制約・要件
（ユーザーから得た非実装的な制約）

### 期待するアウトプット
（ユーザーが明示的に求めた回答形式があればそのまま記載する。例: 「比較表が欲しい」「移行ステップを示して」等）
（ユーザーが特に指定していなければこのセクションは省略し、architect に任せる）

### 参考資料（ユーザー提供）
（ユーザーが明示的に提供したコード・ファイル・資料。なければ省略）
```

## Phase 3: 依頼文の確認

構築した依頼文をユーザーに提示し、承認を得る。

- 依頼文をそのまま表示し「この内容で greenfield-architect に依頼します。修正があれば指摘してください」と伝える
- ユーザーが修正を求めた場合、依頼文を修正して再提示する
- ユーザーが承認したら Phase 4 に進む

## Phase 4: 委託と結果提示

1. Agent ツールで greenfield-architect に委託する
   - prompt: ユーザーが承認した依頼文をそのまま設定する。依頼文にない情報の付与は禁止
2. 結果をそのままユーザーに提示する（自分の意見を加えない）
3. ユーザーが追加の要望を出した場合、内容に応じて適切なフェーズに戻る：
   - 要件自体の変更・追加 → Phase 1
   - 依頼文の修正 → Phase 2
   - 設計の深掘り・別の角度からの検討 → 追加の依頼文を作成し Phase 3 から
