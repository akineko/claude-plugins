# アーキテクチャドキュメントの作成

既存のコードベースや完成した設計を調査し、設計判断の背景と意図を伝えるアーキテクチャドキュメントを作成する。

## 基本原則

このスキルが作るドキュメントは「コードの転記」ではなく「設計の意図と背景の記録」。
コードを読めばわかる事実（ファイル一覧、型定義、API仕様）は書かず、コードからは読み取れない「なぜそうしたか」を残す。

ドキュメントの粒度は「アーキテクチャの方針変更時のみ更新が必要」な抽象度を目指す。コンポーネントの追加やリネームで更新が必要になるなら、それは細かすぎる。

詳細なガイドラインは `references/guidelines.md` にある。判断に迷ったら参照すること。

## ワークフロー概要

```
入力（対象パス or テキスト）
  │
  ├─ Phase 1: 初期化・対象の理解
  │     └─ 入力を判定し、ドキュメント化の対象と範囲を確定
  │
  ├─ Phase 2: コードベース調査
  │     └─ codebase-investigator で構造・パターン・設計意図を調査
  │
  ├─ Phase 3: ドキュメント生成
  │     └─ 調査結果からアーキテクチャドキュメントを作成
  │
  ├─ Phase 4: ユーザー確認・修正
  │     └─ フィードバックに基づく修正ループ
  │
  成果物: 対象ディレクトリ or .claude/tmp/{dir}/ に .md ファイル
```

## Phase 1: 初期化

### 1.1 入力の判定

`$ARGUMENTS` の内容から入力種別を判定する:

| 入力 | 判定方法 | アクション |
|------|---------|-----------|
| ディレクトリパス | 存在するディレクトリ | そのディレクトリを調査対象とする |
| ファイルパス | 存在するファイル（.md等） | 設計書・仕様書として読み込み、関連コードを調査対象とする |
| テキスト | パスでない文字列 | テキストからドキュメント化対象を推定する |
| 空 | — | AskUserQuestion で入力を求める |

### 1.2 成果物パスの決定

以下の優先順位で成果物の配置先を決定する:

1. **プロジェクトのルールを確認**: CLAUDE.md、README、既存のドキュメント構造（`docs/` や `ARCHITECTURE.md` の有無）を軽く調べる
   - 明示的なルールがあれば従う
   - 既存の `ARCHITECTURE.md` や `docs/architecture/` があれば、その配置に揃える
2. **ルールも前例もなければ**:
   - プロジェクト全体が対象 → リポジトリルートに `ARCHITECTURE.md`
   - 特定の機能・領域が対象 → `docs/architecture/{topic}.md`
3. **パスをユーザーに提示して確認する**

### 1.3 ドキュメント化のスコープ確認

対象が広い場合（リポジトリ全体、複数サービスにまたがる等）、AskUserQuestion でスコープを絞る:

```
ドキュメント化の対象が広範囲です。以下のどの範囲を対象としますか？
1. {候補1} — {簡潔な説明}
2. {候補2} — {簡潔な説明}
3. 全体の概観

番号で指定してください。
```

単純なCRUD画面や設計判断のない領域は、ドキュメントを作成する意義が薄い。その場合はユーザーにその旨を伝え、本当に必要か確認する。

## Phase 2: コードベース調査

`codebase-investigator` に調査を依頼する。

### 2.1 調査の方針

調査では「何があるか」だけでなく「なぜそうなっているか」を読み取ることが重要。以下の観点で調査する:

- **構造的な設計判断**: なぜこのレイヤー構成か、なぜこの責務分離か
- **パターンの選択理由**: なぜこのデザインパターンを使っているか、代替は何だったか
- **制約と前提**: 技術的制約、ビジネス要件、パフォーマンス要件など
- **複数レイヤーにまたがる関係性**: データフロー、依存方向、境界の設計意図

### 2.2 入力種別に応じた調査の調整

ファイル入力が設計書（実装設計書、要求仕様書等）の場合、以下の追加観点を調査プロンプトに含める:

- 設計書に記載された設計判断が実際のコードでどう実現されているか
- 設計書の内容と実装の乖離がある場合は、その事実

これにより、一度の調査で設計意図と実装状況の両方を把握できる。

### 2.3 調査の実行

対象範囲に応じて単一または並列で調査する。

#### 単一調査

```
Agent(
  subagent_type="codebase-investigator",
  prompt=下記の調査プロンプト
)
```

#### 並列調査（複数レイヤー/サービスにまたがる場合）

```
Agent(name="research-{領域1}", subagent_type="codebase-investigator", prompt=...)
Agent(name="research-{領域2}", subagent_type="codebase-investigator", prompt=...)
```

#### 調査プロンプト

```
以下の対象についてアーキテクチャドキュメントを作成するための調査を行え。

## 対象
{対象のパスまたは説明}

## 背景情報
{ファイル入力の場合はその内容、テキスト入力の場合はユーザーの説明}

## 調査の観点

以下の4カテゴリに分類して報告せよ。

### 1. 全体構造と責務分離
- レイヤー構成、モジュール構成とその責務
- 各レイヤー/モジュール間の依存方向と通信方法
- 構造上の設計判断とその意図（読み取れる範囲で）

### 2. 設計パターンと選択理由
- 使用しているデザインパターンとその適用箇所
- パターン選択の理由（推測を含む場合は明記）
- 一貫したパターンと例外的な実装

### 3. 制約と前提条件
- 技術的制約（フレームワークの制限、外部サービスの制約等）
- パフォーマンス上の考慮事項
- セキュリティ・認証に関する設計

### 4. データフローと状態管理
- 主要なデータの流れ（概念レベルで）
- 状態管理の戦略と境界
- 外部システムとの統合ポイント

{設計書入力の場合は以下を追加:}
### 5. 設計と実装の整合性
- 設計書の設計判断が実装でどう実現されているか
- 設計書と実装の乖離がある箇所とその内容

注意: 個々のファイル名・関数名・型定義の網羅的な列挙は不要。
構造の「代表例」として必要最小限の具体名に言及するに留めること。
```

## Phase 3: ドキュメント生成

調査結果を元に、ガイドラインに従ってアーキテクチャドキュメントを作成する。

### 3.1 構成

以下のテンプレートを基本とするが、対象に応じてセクションを調整してよい。不要なセクションは省略し、必要なセクションを追加する。

```markdown
# {対象の名称}

## 概要

{この領域が何をするか、1-2文}

## アーキテクチャ概観

{全体構造の概念図や説明。Mermaid図が有効な場合は使用する}

## 設計判断

### {判断1のタイトル}

{なぜこの構成にしたか。トレードオフは何か。他の選択肢を取らなかった理由}

### {判断2のタイトル}

...

## 制約と前提条件

{設計の背景にある技術的制約やビジネス要件}

## 関連ドキュメント

{関連する設計書やドキュメントへのリンク}
```

### 3.2 記述のルール

- **設計判断の「なぜ」を中心に書く**。「何がある」ではなく「なぜそうした」
- **ファイル名・関数名は代表例として**最小限に言及する。網羅的なリストは作らない
- **コード片は含めない**。パターンの説明は概念と責務で行う
- **概念図を活用する**。Mermaid記法で構造やフローを視覚化すると効果的
- **更新頻度を意識する**。「このドキュメントはアーキテクチャの方針が変わったときだけ更新すればよい」という抽象度を保つ

### 3.3 粒度チェック

ドキュメント生成後、以下を自己チェックする。1つでも該当すれば抽象度を上げる:

- [ ] コンポーネントやファイルの追加・リネームで更新が必要になる記述がないか
- [ ] 型定義やインターフェースの転記がないか
- [ ] 個別のAPIエンドポイントやキャッシュキーに言及していないか
- [ ] ディレクトリツリーやファイル一覧を含んでいないか

## Phase 4: ユーザー確認・修正

### 4.1 ドキュメントの提示

生成したドキュメントの概要を簡潔に提示する:

- 記載した設計判断の項目タイトルを箇条書き
- ドキュメントの全体的な構成

提示後、AskUserQuestion で確認:

```
アーキテクチャドキュメントを作成しました: {成果物パス}

内容を確認して、修正したい箇所があれば指摘してください。
問題なければ「OK」と回答してください。
```

### 4.2 フィードバック対応

| ユーザーの回答 | アクション |
|---|---|
| 「OK」等 | Phase 5 へ |
| 修正指示あり | 指示に基づいて修正し、再度 4.1 に戻る |
| 追加調査が必要 | Phase 2 の追加調査を実行してから修正 |

修正時も粒度チェック（3.3）を実施する。

## Phase 5: 完了

1. 成果物のファイルパスを案内する
2. ドキュメントの概要（記載した設計判断のポイント）を3-5行で提示する
3. メンテナンスの指針を案内する: 「アーキテクチャの方針が変わったときに更新してください。ファイルの追加・リネーム程度では更新不要です」
