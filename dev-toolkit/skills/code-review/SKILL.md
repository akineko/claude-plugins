# Code Review

3つの専門レビュアーエージェントが異なる視点でコードをレビューし、統合した改善項目を作成する。

## レビュアー構成

| レビュアー | 担当領域 | サブエージェント |
|-----------|---------|----------------|
| 正確性 | ロジック破綻、境界条件、例外系、型安全性 | `correctness-reviewer` |
| 設計 | 責務分離、命名、構造、変更容易性 | `architecture-reviewer` |
| 性能 | パフォーマンス、スケーラビリティ、リソース効率、障害解析性 | `performance-reviewer` |

セキュリティ専門のレビュアーは含まない。ただし、各レビュアーが自身の専門領域の観点から自然に発見したセキュリティ上の問題は指摘してよい。

## ワークフロー

### Step 1: レビュー対象の特定

ユーザーの指示からレビュー対象を特定し、エージェントに渡すコンテキスト情報を構成する。
差分やコードの内容自体をプロンプトに埋め込む必要はない — エージェントが自律的に取得・確認する。

#### 対象の特定

##### 明示的な指定がある場合

| 指定方法 | 対象種別 |
|---------|---------|
| ステージング済み / 未ステージング / コミット指定 / ブランチ差分 | diff系 |
| PR指定 | PR |
| ディレクトリ/ファイル指定 | ファイル直接 |

##### 指定がない場合の自動検出

以下の順で確認し、最初に差分が見つかったものを使う：

1. `git diff --staged` — ステージング済みの変更
2. `git diff` — 未ステージングの変更
3. `git diff HEAD~1 HEAD` — 直近コミット

#### コンテキスト情報の構成

対象種別に応じて、エージェントが自律的にコードを確認するために必要十分な情報を構成する。

##### diff系の場合

`git diff --stat ...` 等で変更ファイル一覧を取得し、以下を構成する：

- 変更ファイル一覧（ファイルパスと変更行数）
- 差分を取得するための git コマンド

##### PR の場合

`gh pr view <number>` で PR 情報を取得し、以下を構成する：

- PR 番号とタイトル
- 差分を取得するためのコマンド（`gh pr diff <number>`）
- PR の説明を確認するためのコマンド（`gh pr view <number>`）

##### ファイル/ディレクトリ指定の場合

- 対象のファイルパスまたはディレクトリパス

### Step 2: 3つのレビュアーエージェントを並列起動

3つのレビュアーを同時に起動することで、レビュー時間を短縮し、各レビュアーが互いの結果に影響されない独立した視点を確保する。

#### 2-1. 共通プロンプトを組み立てる

Step 1 で構成したコンテキスト情報を埋め込んだ共通プロンプトを組み立てる。各エージェントのレビュー観点・方針はエージェント定義に従うため、スキルのプロンプトではレビュー対象の情報提供と出力形式の指定のみ行う。

````
以下のコードをレビューしてください。

## レビュー対象
{review_context}

## 指示
1. 上記の情報をもとに、対象コードと変更内容を確認する
2. 必要に応じて関連ファイル（import先、呼び出し元など）も確認する
3. あなたの専門領域の観点でレビューする

## 出力形式
エージェント定義の出力形式ではなく、以下のJSON形式で出力すること（マークダウンのコードブロックで囲む）：
```json
{
  "reviewer": "{レビュアー名}",
  "findings": [
    {
      "severity": "high" | "medium" | "low",
      "category": "指摘のカテゴリ",
      "file": "対象ファイルパス",
      "line": "対象行（範囲可、例: 42-50）",
      "issue": "何が問題か",
      "suggestion": "どう改善すべきか",
      "reason": "なぜそれが問題なのか"
    }
  ],
  "summary": "このレビュアー視点での全体的な所見（1-2文）"
}
```

指摘がない場合は findings を空配列にし、summary にその旨を記載する。
````

##### `{review_context}` の構成例

**diff系:**
```
ステージング済みの変更（3ファイル）

### 変更ファイル
- src/auth/login.ts (+45, -12)
- src/auth/types.ts (+8, -0)
- src/middleware/auth.ts (+22, -18)

### 差分の確認方法
`git diff --staged` を実行してください。
```

**PR:**
```
PR #123「ログイン機能の追加」

### 確認方法
- 差分: `gh pr diff 123`
- PR説明: `gh pr view 123`
```

**ファイル/ディレクトリ:**
```
以下のファイルを直接レビューしてください。

### 対象
- src/features/Auth/
```

#### 2-2. 3つのエージェントを1回のレスポンスで同時に起動する

上記プロンプトを使い、以下の3つの Agent tool call を **同一レスポンス内で** 発行する。
1つ目のエージェントの結果を待ってから2つ目を起動する、といった逐次的な起動はしない。

| # | subagent_type | 観点 |
|---|---------------|------|
| 1 | `correctness-reviewer` | 正確性 |
| 2 | `architecture-reviewer` | 設計 |
| 3 | `performance-reviewer` | 性能 |

### Step 3: レビュー結果の統合

3つのエージェントから返ってきた結果を以下の手順で統合する。

#### 3-1. 重複・関連の検出

同じファイル・行に複数のレビュアーが言及している場合：
- 同じ問題を異なる視点で指摘 → 1つにまとめ、全視点を併記
- 異なる問題を指摘 → それぞれ独立した指摘として残す

#### 3-2. 矛盾の解決

レビュアー間で矛盾する指摘がある場合：
- 矛盾の内容と各レビュアーの根拠を併記する
- オーケストレーターとしての推奨判断を付記する
- 最終判断はユーザーに委ねる

#### 3-3. severity の扱い

- 各レビュアーの報告をそのまま尊重する（統合時に勝手に変更しない）
- 複数レビュアーが同じ箇所を指摘している場合は、その旨を明記する（severity自体は変えない）

### Step 4: 総合評価の判定

| 評価 | 条件 |
|------|------|
| **Approve** | high: 0、medium: 0 |
| **Request Changes** | high: 1以上 |
| **Needs Discussion** | レビュアー間で矛盾する指摘がある、または判断が難しい medium がある |

### Step 5: 最終レポートの出力

```markdown
# Code Review Report

## 概要
- **レビュー対象**: {対象の説明（例: ステージング済みの変更、PR #123、src/features/Auth/）}
- **対象ファイル数**: {N}
- **指摘件数**: High {n} / Medium {n} / Low {n}
- **総合評価**: {Approve / Request Changes / Needs Discussion}

## 指摘事項

### High（対応推奨）

#### 1. {issue の要約}
- **ファイル**: `{file}:{line}`
- **視点**: {レビュアー名}（複数の場合はすべて記載）
- **問題**: {issue}
- **理由**: {reason}
- **改善案**: {suggestion}

### Medium（検討推奨）
（同様の形式）

### Low（任意対応）
（同様の形式）

## レビュアー間の見解の相違
（矛盾する指摘がある場合のみ記載）

#### 1. {対象箇所}
- **{レビュアーA}の見解**: ...
- **{レビュアーB}の見解**: ...
- **推奨**: {オーケストレーターの判断}

## 各レビュアーの総評

| レビュアー | 総評 |
|-----------|------|
| 正確性 | {summary} |
| 設計 | {summary} |
| 性能 | {summary} |

## 複数レビュアーの指摘が重複・関連する箇所
（同じファイル・行に対して複数のレビュアーが指摘している場合、ここでまとめる）
```

#### 出力時の注意

- 指摘が0件の場合は「指摘事項なし」と明記し、各レビュアーの総評のみ表示する
- 見解の相違がない場合は該当セクションを省略する
- 重複・関連がない場合は該当セクションを省略する

## 禁止事項

- 直接のコードレビュー → 必ず3つのレビュアーに委譲する
- コードの修正 → レビュー結果の報告のみ行う
- レビュアーの逐次起動 → 3つのAgent tool callは必ず同一レスポンスで発行する。1つ起動して結果を待ってから次を起動するのは禁止

## 使用例

```
/code-review                    # 自動検出（staged → unstaged → last commit）
/code-review --staged           # ステージ済みの変更
/code-review HEAD~1             # 直前のコミット
/code-review main...HEAD        # ブランチの全変更
/code-review src/features/Auth/ # 特定ディレクトリ
/code-review 123                # PR #123
```
