# Code Review

3つの専門レビュアーエージェントが異なる視点でコードをレビューし、統合した改善項目を作成する。

## レビュアー構成

| レビュアー | 担当領域 | サブエージェント |
|-----------|---------|----------------|
| 正確性 | ロジック破綻、境界条件、例外系、型安全性 | `correctness-reviewer` |
| 設計 | 責務分離、命名、構造、変更容易性 | `architecture-reviewer` |
| 性能 | パフォーマンス、スケーラビリティ、リソース効率、障害解析性 | `performance-reviewer` |

セキュリティ専門のレビュアーは含まない。ただし、各レビュアーが自身の専門領域の観点から自然に発見したセキュリティ上の問題は指摘してよい。

## ワークフロー

### Step 1: レビュー対象の差分を取得

ユーザーの指示に応じて差分を取得する。

#### 明示的な指定がある場合

| 指定方法 | コマンド |
|---------|---------|
| ステージング済み | `git diff --staged` |
| 未ステージング | `git diff` |
| コミット指定 | `git diff <commit>~1 <commit>` |
| ブランチ差分 | `git diff <base>...<head>` |
| PR指定 | `gh pr diff <number>` |
| ディレクトリ/ファイル | 対象を直接読み取り（diffなし） |

#### 指定がない場合の自動検出

以下の順で確認し、最初に差分が見つかったものを使う：

1. `git diff --staged` — ステージング済みの変更
2. `git diff` — 未ステージングの変更
3. `git diff HEAD~1 HEAD` — 直近コミット

#### サイズ制限

差分が500行を超える場合は、ユーザーに対象を絞るか確認する。

### Step 2: 3つのレビュアーエージェントを並列起動

Agent ツールで3つのレビュアーを **1回のメッセージで同時に** 起動する。

各エージェントには以下を渡す：
- レビュー対象のdiff（Step 1で取得したもの）
- 対象ファイルの読み取り指示（diffだけでなく、関連するコードの文脈も確認するため）

#### プロンプト

各エージェントのレビュー観点・方針はエージェント定義に従う。スキルのプロンプトでは差分の提供と出力形式の指定のみ行う。

```
以下のコードをレビューしてください。

## レビュー対象の差分
{diff}

## 指示
1. 差分に含まれるファイルを Read ツールで読み、変更箇所の前後の文脈を把握する
2. 必要に応じて関連ファイル（import先、呼び出し元など）も確認する
3. あなたの専門領域の観点でレビューする

## 出力形式
エージェント定義の出力形式ではなく、以下のJSON形式で出力すること（マークダウンのコードブロックで囲む）：
```json
{
  "reviewer": "{レビュアー名}",
  "findings": [
    {
      "severity": "high" | "medium" | "low",
      "category": "指摘のカテゴリ",
      "file": "対象ファイルパス",
      "line": "対象行（範囲可、例: 42-50）",
      "issue": "何が問題か",
      "suggestion": "どう改善すべきか",
      "reason": "なぜそれが問題なのか"
    }
  ],
  "summary": "このレビュアー視点での全体的な所見（1-2文）"
}
```

指摘がない場合は findings を空配列にし、summary にその旨を記載する。
```

起動するサブエージェント：
1. `correctness-reviewer` — 正確性の観点
2. `architecture-reviewer` — 設計の観点
3. `performance-reviewer` — 性能の観点

### Step 3: レビュー結果の統合

3つのエージェントから返ってきた結果を以下の手順で統合する。

#### 3-1. 重複・関連の検出

同じファイル・行に複数のレビュアーが言及している場合：
- 同じ問題を異なる視点で指摘 → 1つにまとめ、全視点を併記
- 異なる問題を指摘 → それぞれ独立した指摘として残す

#### 3-2. 矛盾の解決

レビュアー間で矛盾する指摘がある場合：
- 矛盾の内容と各レビュアーの根拠を併記する
- オーケストレーターとしての推奨判断を付記する
- 最終判断はユーザーに委ねる

#### 3-3. severity の扱い

- 各レビュアーの報告をそのまま尊重する（統合時に勝手に変更しない）
- 複数レビュアーが同じ箇所を指摘している場合は、その旨を明記する（severity自体は変えない）

### Step 4: 総合評価の判定

| 評価 | 条件 |
|------|------|
| **Approve** | high: 0、medium: 0 |
| **Request Changes** | high: 1以上 |
| **Needs Discussion** | レビュアー間で矛盾する指摘がある、または判断が難しい medium がある |

### Step 5: 最終レポートの出力

```markdown
# Code Review Report

## 概要
- **レビュー対象**: {diff取得方法の説明}
- **対象ファイル数**: {N}
- **指摘件数**: High {n} / Medium {n} / Low {n}
- **総合評価**: {Approve / Request Changes / Needs Discussion}

## 指摘事項

### High（対応推奨）

#### 1. {issue の要約}
- **ファイル**: `{file}:{line}`
- **視点**: {レビュアー名}（複数の場合はすべて記載）
- **問題**: {issue}
- **理由**: {reason}
- **改善案**: {suggestion}

### Medium（検討推奨）
（同様の形式）

### Low（任意対応）
（同様の形式）

## レビュアー間の見解の相違
（矛盾する指摘がある場合のみ記載）

#### 1. {対象箇所}
- **{レビュアーA}の見解**: ...
- **{レビュアーB}の見解**: ...
- **推奨**: {オーケストレーターの判断}

## 各レビュアーの総評

| レビュアー | 総評 |
|-----------|------|
| 正確性 | {summary} |
| 設計 | {summary} |
| 性能 | {summary} |

## 複数レビュアーの指摘が重複・関連する箇所
（同じファイル・行に対して複数のレビュアーが指摘している場合、ここでまとめる）
```

#### 出力時の注意

- 指摘が0件の場合は「指摘事項なし」と明記し、各レビュアーの総評のみ表示する
- 見解の相違がない場合は該当セクションを省略する
- 重複・関連がない場合は該当セクションを省略する

## 禁止事項

- 直接のコードレビュー → 必ず3つのレビュアーに委譲する
- コードの修正 → レビュー結果の報告のみ行う

## 使用例

```
/code-review                    # 自動検出（staged → unstaged → last commit）
/code-review --staged           # ステージ済みの変更
/code-review HEAD~1             # 直前のコミット
/code-review main...HEAD        # ブランチの全変更
/code-review src/features/Auth/ # 特定ディレクトリ
/code-review 123                # PR #123
```
