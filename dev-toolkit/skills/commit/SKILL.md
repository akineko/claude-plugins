# Commit

ステージング済みの変更を確認し、Conventional Commits 形式のコミットメッセージを生成して自動的にコミットする。

## ワークフロー

### Step 1: ステージング状態の確認

`git diff --staged` でステージング済みの変更を確認する。

- ステージング済みの変更がない場合 → ユーザーにその旨を伝えて終了する
- `.env`、credentials、秘密鍵などの機密ファイルが含まれている場合 → ユーザーに警告して確認を取る

### Step 2: コミットメッセージの生成

変更内容を分析し、以下の形式でコミットメッセージを生成する。

#### タイトル（1行目）

Conventional Commits 形式で記述する。

```
<type>(<scope>): <summary>
```

##### type

変更の性質に応じて選択する。

| type | 用途 |
|------|------|
| `feat` | 新機能の追加 |
| `fix` | バグ修正 |
| `refactor` | 機能変更を伴わないコード改善 |
| `docs` | ドキュメントの変更 |
| `test` | テストの追加・修正 |
| `chore` | ビルド、CI、依存関係などの雑務 |
| `perf` | パフォーマンス改善 |
| `style` | コードスタイルの変更（フォーマット、セミコロン等） |
| `ci` | CI/CD 設定の変更 |

##### scope

monorepo の場合のみ付与する。CLAUDE.md やディレクトリ構造（`packages/`、`apps/`、ワークスペース設定等）からワークスペースやパッケージの名前を推測し、変更が単一のスコープに収まる場合に付与する。複数スコープにまたがる場合や monorepo でない場合は省略する。

##### summary

- 変更の本質を簡潔に表現する（目安50文字以内）
- **何をしたか（what）** ではなく **何を実現するか** を書く

#### ボディ（3行目以降）

変更の **理由（why）** を箇条書きで簡潔に記述する。

```
- 理由1
- 理由2
```

箇条書きだけでは伝わりにくい背景や判断の経緯がある場合は、箇条書きの後に空行を挟んで文章で補足する。

```
- 理由1
- 理由2

補足の文章をここに書く。
```

##### ボディを省略するケース

以下のような変更は理由が自明なためボディを省略してよい：

- typo 修正
- フォーマット修正
- 単純なドキュメント更新

### Step 3: コミットの実行

生成したコミットメッセージを使い `git commit` を実行する。コミットメッセージは HEREDOC で渡してフォーマットを保持する。

```bash
git commit -m "$(cat <<'EOF'
<type>(<scope>): <summary>

- why 1
- why 2
EOF
)"
```

### Step 4: 結果の報告

コミット完了後、以下をユーザーに伝える：

- コミットメッセージ全文
- 変更ファイルの一覧（`git diff --stat` 相当の情報）

## 禁止事項

- ステージング内容の変更（`git add` / `git reset` の実行）
- コミット前のユーザー確認（確認なしで即コミットする）
- リモートへの push
