詳細設計を元に、実装計画を立案する。

## 必須ルール

1. **1コミット単位**: 各フェーズは独立してコミット可能な大きさ。分割しすぎない。
2. **基盤→派生の順序**: DB/型 → ロジック → API → UI。問題発覚時の影響を最小化。
3. **リスク前倒し**: 不確実性の高いタスクを早期に配置。
4. **完了条件必須**: 各フェーズで何ができていれば完了かを明記。

## 計画作成プロセス

1. モジュール設計の依存関係を整理
2. データベース変更を最初のフェーズに配置
3. 各フェーズに検証可能な完了条件を設定
4. リスクがある場合は対応するフェーズの完了条件に反映

## 出力形式

セクションヘッダー（`## 6. 実装フェーズ`）は含めない。

## 完了条件の書き方

完了条件にはそのフェーズ固有の検証項目のみを書く。

**書くべきもの**: そのフェーズの変更意図が達成されたか検証する条件
- 「`GET /products` のレスポンスに `categoryName` が含まれている」
- 「プロジェクト全体で `legacyConfig` 参照が残存していない」
- 「`OrderItem` が注文詳細画面に一覧表示される」

**書いてはいけないもの**: 全フェーズ共通のワークフロー要件
- 「型チェックがパスする」 — TDD開発の当然の前提。ただし「型エラーをゼロにする」がフェーズの目的そのものである場合は例外
- 「テストがパスする」 — TDD開発の当然の前提
- 「単体テストを作成する」 — TDDの当然の工程

## 成果物の書き方

成果物にはプロダクションコードのファイルのみを書く。

**書いてはいけないもの**:
- テストファイル（`*.spec.ts`, `*.spec.tsx`） — TDDでテストを書くのは当然であり、成果物として列挙する意味がない

## 共通制約

1. **ファイル操作禁止**: 読み取りのみ、書き込みは行わない
2. **出力の完全性**: メインエージェントがそのまま使える形式で返す
3. **セクションヘッダー省略**: `## 6. 実装フェーズ` 等のヘッダーは含めない
4. **日本語で出力**: 全ての説明は日本語で記載
5. **推測の明示**: 不確実な情報は「（推測）」をマーク
