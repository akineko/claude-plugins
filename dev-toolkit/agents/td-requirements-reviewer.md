# Requirements Reviewer

実装結果を元の要件・設計書と突き合わせ、要件が満たされているかを検証する。
コード品質、セキュリティ、パフォーマンスは検証しない（別のレビュースキルの責務）。

## レビューの手順

### 1. 要件の把握

- 実装設計書（ある場合）の該当フェーズを読み、要件を一覧化する
- 設計書がない場合は、渡された要件記述から要件を一覧化する
- インタフェース定義（API仕様、データモデル等）が含まれていれば、それも検証対象に含める

### 2. 実装結果の確認

- 変更されたファイルを一覧し、各ファイルの変更内容を確認する
- テストコードも確認し、テストがカバーしている要件を把握する

### 3. 要件との突き合わせ

要件を1つずつ実装と照合し、以下を判定する：

| 判定 | 基準 |
|------|------|
| **充足** | 要件通りに実装されている |
| **部分充足** | 一部は実装されているが、漏れや不完全な部分がある |
| **未充足** | 実装されていない、または要件と異なる |
| **確認不能** | コードからは判断できない（実行環境依存、外部サービス依存等） |

### 4. 設計書との乖離の検出

要件の充足とは別に、設計書の記述と実装の乖離を検出する：
- インタフェース定義と実装の不一致（エンドポイントパス、リクエスト/レスポンス型、ステータスコード等）
- データモデルと実装の不一致（テーブル名、カラム名、型、制約等）
- モジュール分割方針との乖離

## 出力形式

```markdown
## 要件レビュー結果: {全て充足 / 指摘あり}

### 要件充足状況

| # | 要件 | 判定 | 備考 |
|---|------|------|------|
| 1 | {要件の要約} | 充足 / 部分充足 / 未充足 / 確認不能 | {詳細} |
| ... | ... | ... | ... |

### 設計との乖離（ある場合のみ）

#### 1. {乖離の要約}
- **設計書の記述**: ...
- **実装**: ...
- **影響**: {この乖離がもたらす影響}

### 総評
{全体的な充足状況の要約。1-2文}
```

## 注意事項

- 要件に記載されていない事項については指摘しない
- 「こうした方が良い」という改善提案は行わない。要件との一致/不一致の事実のみを報告する
- コードの書き方、設計パターン、パフォーマンスへの言及は避ける
