# スキルビルダー

ユーザーとの対話を通じて、ベストプラクティスに従ったClaudeスキルを作成するためのスキル。

## 基本方針

- ユーザーの意図を正確に把握してからスキルを作成する
- 作成するスキルの文章はユーザーの言語に合わせる
- スキルの配置先は `.claude/skills/<スキル名>/` ディレクトリ
- 指示は「〜を適切に処理する」ではなく「〜の場合は〜する」と具体的に書く
- 「なぜ」を説明する — Claudeは理由が分かると機械的な遂行を超えた結果を出す
- シンプルに始める — 参照ファイルやアセットは本当に必要な場合のみ追加する
- コンテキストウィンドウは共有資源 — SKILL.md本文は500行/5,000語以内に抑え、詳細はreferences/に分離する
- 使用例を必ず含める
- エラーハンドリングを含める
- 重要な指示はSKILL.mdの上部に配置する

## 手順

### ステップ0：リクエストの分類

ユーザーのリクエストを以下に分類し、対応するステップに進む：

A. **新規作成**: 「スキルを作りたい」「〇〇を自動化したい」→ ステップ1Aへ
B. **会話のスキル化**: 「これをスキルにして」「今のワークフローをスキル化して」→ ステップ1Bへ

### ステップ1A：ユースケースの把握（新規作成）

まずユーザーに以下を確認する。会話の文脈からすでに分かっている情報は改めて聞かない。質問はできるだけ1〜2回の対話にまとめ、一度に聞く質問は3問以内にする。

必ず確認する項目（この順序で進める）：
1. このスキルで何を達成したいか（目的）
2. 想定される具体的な使用シーン（2〜3個）— 目的が明確になってから聞く
3. ユーザーがどのような言葉でこのスキルを呼び出すか（トリガーフレーズ）— 使用シーンから自然に導出できる場合は省略可

目的が抽象的な場合（「効率化したい」「自動化したい」など）は、具体的な入力と出力の例を1つ挙げてもらう。

目的が明確になった後、以下の条件に該当する場合は追加確認する：
- 既存スキルと機能が重なりそうな場合 → 起動すべきでない場面を確認
- 外部サービス連携が言及された場合 → MCPサーバーの必要性を確認
- 出力形式が自明でない場合 → 形式を確認（ドキュメント、コード、ワークフロー実行など）
- 繰り返し処理やデータ変換が含まれる場合 → スクリプトの必要性を確認
- 定型的な出力が求められる場合 → テンプレートやアセットの必要性を確認

**ステップ2に進む前に：**
把握した内容を簡潔に要約し、ユーザーに確認する。認識のズレがあればここで修正する。

### ステップ1B：会話履歴の抽出（スキル化）

会話履歴からワークフローを抽出する。以下の情報を特定する：
- 使用したツール — スキルのツール依存を把握するため
- ステップの順序 — 再現可能な手順を構成するため
- ユーザーが行った修正 — 初回で正しく動かなかった箇所＝明示的な指示が必要な箇所
- 入出力フォーマット — スキルの期待値を定義するため

抽出結果をユーザーに確認し、不足があれば補完する。会話履歴が不十分な場合は、スキル化したいワークフローの具体的な手順をユーザーに説明してもらう。

**ステップ2に進む前に：**
把握した内容を簡潔に要約し、ユーザーに確認する。認識のズレがあればここで修正する。

### ステップ2：スキル設計の決定

ユースケースに最適なパターンを `references/patterns.md` から選定する：
- 順序実行型 / 反復改善型 / コンテキスト判定型 / ドメイン知識型 / マルチMCP連携型

パターンを選定したら、選定理由とともにユーザーに提示し、合意を得てから次のステップに進む。

### ステップ3：フォルダ構成の設計

`references/skill-structure-guide.md` を参照して、スキルのフォルダ構成を設計する。

基本構成：
```
スキル名/
├── SKILL.md          # 必須
├── scripts/          # 任意 - 実行可能コード
├── references/       # 任意 - 必要時に読み込むドキュメント
└── assets/           # 任意 - テンプレート、フォント、アイコンなど
```

スクリプトを含める場合は `references/skill-structure-guide.md` を参照し、以下の順で判断する：
1. **活用判断** — そのスクリプトは本当に必要か？マークダウン指示で十分な処理にスクリプトは不要
2. **言語選択** — プロジェクトの採用言語をデフォルトとする。別言語を使う場合はユーザーに理由を提示し承認を得る

### ステップ4：SKILL.mdの作成

#### フロントマター

`references/yaml-reference.md` のルールを厳守する。

```yaml
---
name: kebab-case-の名前
description: >
  何をするか。いつ使うか。トリガーフレーズ。
  否定トリガー（必要な場合）。
---
```

**descriptionの重要ルール：**
- 「何をするか」と「いつ使うか」の両方を必ず含める
- ユーザーが実際に言いそうなトリガーフレーズを含める
- Claudeはスキルの起動に消極的になりがちなので、やや積極的に書く
- 類似スキルとの混同リスクがある場合は否定トリガーを追加する

#### 本文テンプレート

```markdown
# スキル名

## 概要
このスキルの目的と価値を簡潔に説明。

## 手順

### ステップ1：[最初の作業]
具体的で実行可能な指示を記述。
期待される結果を明記。

### ステップ2：[次の作業]
（必要なステップ数だけ繰り返す）

## 使用例

### 例1：[よくあるシナリオ]
ユーザーの発言：「〜してほしい」
実行内容：
1. ...
2. ...
結果：...

## トラブルシューティング

### エラー：[よくあるエラー]
原因：...
解決策：...
```

#### 指示文のベストプラクティス

- **命令形を使う**（「コマンドを実行する」「出力を確認する」）
- **具体的かつ実行可能に** — 「データを検証する」ではなく、何をチェックし失敗時にどうするかを記述する
- **「なぜ」を説明する** — Claudeは理由の説明に良く反応する。機械的なルールよりも重要な理由を伝える
- **例を含める** — 期待される動作を伝える最も効果的な方法の一つ
- **段階的開示を活用する** — コア指示はSKILL.mdに、詳細な参考資料は `references/` に配置する

### ステップ5：品質チェック（ファイル書き出し前）

草案を `references/checklist.md` の基準に照らしてセルフチェックする。問題があれば修正する。チェックを完了するまでステップ6に進んではならない。

### ステップ6：成果物の提出

`Write` ツールで `.claude/skills/<スキル名>/` にファイルを作成する。

1. SKILL.mdを書き出す
2. 必要に応じて `references/`、`scripts/`、`assets/` 配下のファイルも作成する
3. 作成後、`Read` ツールで内容を確認する

### ステップ7：テストケースの提案

2〜3個の現実的なテストプロンプトを提案する。

```
起動すべきケース：
- 「[自然なリクエスト1]」
- 「[言い換えたリクエスト2]」
- 「[エッジケースのリクエスト3]」

起動すべきでないケース：
- 「[無関係なクエリ1]」
- 「[似ているが対象外のクエリ2]」
```

デバッグ手法：Claudeに「[スキル名]のスキルをいつ使いますか？」と質問すると、descriptionを引用して回答するので、トリガー精度の確認に使える。

スキル運用後にトリガー不足・過剰トリガー・指示の不遵守などの問題が発生した場合は `references/troubleshooting.md` を参照する。
