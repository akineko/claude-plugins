# トラブルシューティングガイド

スキル開発でよくある問題と解決策。

---

## 1. ファイル構造のエラー

### SKILL.mdが見つからない

**原因：** ファイル名が正確に `SKILL.md` でない（大文字小文字を区別する）。

**解決策：** `SKILL.md` にリネームする。`skill.md`、`SKILL.MD`、`Skill.md` などのバリエーションは不可。

### フロントマターが無効

**原因：** YAMLフォーマットの問題。

**よくあるミスと修正：**

デリミタの欠落 — フロントマターは `---` で囲む必要がある：
```yaml
# 誤り
name: my-skill
description: 何かをする

# 正しい
---
name: my-skill
description: 何かをする
---
```

descriptionに特殊文字が含まれる場合 — コロン、引用符などのYAML特殊文字がある場合は、引用符で囲むか `>` ブロックスカラーを使用する：
```yaml
# 安全な方法
description: >
  データファイルを分析する。ユーザーが「データ分析」や
  CSV処理について言及した場合に使用する。
```

### スキル名が無効

**原因：** 名前にスペース、大文字、またはアンダースコアが含まれている。

```yaml
# 誤り
name: My Cool Skill
name: my_cool_skill

# 正しい
name: my-cool-skill
```

## 2. トリガーの問題

### スキルが自動で読み込まれない（トリガー不足）

**診断チェックリスト：**
- descriptionが一般的すぎないか？
- ユーザーが実際に言いそうなトリガーフレーズが含まれているか？
- 関連するファイル形式が記載されているか？

**修正方法：**
1. Claudeに「[スキル名]スキルはいつ使いますか？」と聞く。Claudeはdescriptionを引用するので、何が不足しているかが分かる。
2. より具体的なトリガーフレーズとコンテキストを追加する。
3. Claudeはスキルの起動に消極的な傾向があるため、descriptionをやや積極的に書く。

**修正前後の例：**
```yaml
# 修正前（トリガー不足）
description: データ分析を支援する。

# 修正後（確実にトリガーする）
description: >
  CSVおよびExcelデータファイルの統計分析。回帰分析、クラスタリング、
  可視化に対応。ユーザーが.csvや.xlsxファイルをアップロードした場合、
  「データ分析」「統計」「相関」「トレンド」と言った場合、
  または表形式データからグラフを作成したい場合に使用する。
```

### 無関係なクエリでスキルが読み込まれる（過剰トリガー）

**修正方法：**

1. 否定トリガーを追加する：
```yaml
description: >
  CSVファイルの高度な統計モデリング。回帰分析、クラスタリング、
  仮説検定に使用する。単純なデータ探索や基本的なグラフ作成には
  使用しない（data-vizスキルを代わりに使う）。
```

2. ドメインをより具体的にする：
```yaml
# 広すぎる
description: ドキュメントを処理する

# 具体的
description: >
  PDF形式の法律文書を契約レビューおよび条項抽出のために処理する。
```

## 3. 指示の遵守

### Claudeが指示に従わない

**よくある原因と解決策：**

**指示が冗長すぎる：**
- 指示は簡潔でスキャンしやすくする
- 順序立てた手順には番号付きリストを使用する
- 詳細な参考資料は `references/` ファイルに移動する

**重要な指示が埋もれている：**
- 最も重要な指示をSKILL.mdの上部に配置する
- `## 重要` や `## 必須` ヘッダーを使用する
- 見落とされがちな重要ポイントは繰り返す

**曖昧な表現：**
```markdown
# 曖昧（Claudeが自由に解釈する可能性あり）
データを適切にバリデーションしてください。

# 具体的（Claudeが正確に何をすべきか分かる）
create_projectを呼び出す前に以下を確認する：
- プロジェクト名が空でないこと
- 少なくとも1人のチームメンバーがアサインされていること
- 開始日が過去の日付でないこと
いずれかのチェックに失敗した場合、処理を進める前にユーザーに入力の修正を求める。
```

**硬直的なルールへの過度な依存：**
「絶対にXをしろ」「決してYをするな」ではなく、Xが重要な理由やYを避けるべき理由を説明する。Claudeは命令よりも理由付けに良く反応する。

**複雑なタスクでのモデルの「手抜き」：**
ユーザープロンプトに以下を追加する（SKILL.md内よりも効果的）：
```
丁寧に時間をかけて取り組んでください。
速度よりも品質を重視してください。
バリデーションステップを省略しないでください。
```

## 4. MCP連携

### スキルは読み込まれるがMCP呼び出しが失敗する

**チェックリスト：**

1. MCPサーバーの接続を確認する：
   - `claude mcp list` コマンドで登録済みサーバーを確認する
   - `.claude/settings.json` または `~/.claude.json` でMCPサーバー設定を確認する

2. MCPサーバーの起動状態を確認する

3. MCPを単独でテストする（スキルなしで）：
   - Claude Code上で直接MCPツールを呼び出して動作を確認する
   - これが失敗する場合、問題はスキルではなくMCP接続にある

4. スキル内のツール名がMCPサーバーの実際のツール名と一致しているか確認する（大文字小文字を区別する）

## 5. パフォーマンスとコンテキスト

### スキルが遅く感じる、またはレスポンス品質が低下する

**原因：**
- SKILL.mdの内容が大きすぎる
- `.claude/skills/` 配下のスキル数が多すぎる
- 段階的開示を使わず全コンテンツがインラインで読み込まれている

**解決策：**

1. SKILL.mdのサイズを最適化する：
   - 詳細なドキュメントを `references/` に移動する
   - インラインではなく参照へのリンクと、いつ読むべきかのガイダンスを設ける
   - 目安：SKILL.mdは500行 / 約5000語以内

2. スキル数を見直す：
   - `.claude/skills/` 配下に大量のスキルフォルダがないか確認する
   - 関連する機能を1つのスキルにまとめることを検討する
   - 現在のプロジェクトに不要なスキルは別の場所に退避する

3. 段階的開示を適切に使用する：
   - フロントマター：最小限、トリガー判定のみ
   - SKILL.md本文：コアワークフローの指示
   - references/：詳細ドキュメント、APIガイド、大規模なサンプル

4. 同時に有効なスキルの数を見直す（20〜50以上を同時に有効にしていないか）
