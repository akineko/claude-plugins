# スキル構成ガイド

## フォルダ構成の詳細

```
スキル名/
├── SKILL.md          # 必須 - メインのスキルファイル
├── scripts/          # 任意 - 実行可能コード
│   ├── process_data.py
│   └── validate.sh
├── references/       # 任意 - 必要時に読み込むドキュメント
│   ├── api-guide.md
│   └── examples/
└── assets/           # 任意 - テンプレート、フォント、アイコンなど
    └── report-template.md
```

## SKILL.md（必須）

メインのスキルファイル。YAMLフロントマターとMarkdownのインストラクションで構成される。

命名ルール：
- 必ず `SKILL.md` と正確に記述する（大文字小文字を区別）
- `SKILL.MD`、`skill.md`、`Skill.md` などのバリエーションは受け付けない

## scripts/ ディレクトリ（任意）

実行可能なコード（Python、Bash等）を格納する。

用途：
- データのバリデーションや前処理
- 出力の品質チェック
- 決定論的な処理の確実な実行

スクリプト活用の判断基準：
- バリデーション（入力値チェック、フォーマット検証）→ スクリプト推奨。コードは確定的だが、言語指示の解釈はブレる
- データ加工（CSV変換、画像処理など）→ スクリプト推奨
- ワークフローの判断や文章生成 → マークダウン指示で十分
- 参考：既存の公開スキル（docx、pptx、xlsx）はスクリプトで品質を担保するパターンの好例

### スクリプト言語の選択

**デフォルト：プロジェクトの採用言語を使う。** スキルはプロジェクト内（`.claude/skills/`）に配置されるため、スクリプトもプロジェクトと同じ言語エコシステムに属すべきである。言語が揃っていれば、チームメンバーが読める・保守できる・既存のツールチェーンでテストできる。

言語選択の手順：
1. プロジェクトの `package.json`、`pyproject.toml`、`go.mod` 等を確認し、主要言語を特定する
2. そのスクリプトがプロジェクト言語で実現可能かを判断する
3. プロジェクト言語で実現可能 → その言語で書く
4. プロジェクト言語では困難 → 別言語を採用する理由をユーザーに提示し、承認を得てから進める

別言語が妥当なケース（承認は必須）：
- シェル操作のラッパー（パイプライン、プロセス制御）→ Bash
- プロジェクト言語にライブラリが存在しない専門処理（画像処理、科学計算等）→ 適切な言語
- 1行で済む単純なコマンド呼び出し → Bash

## references/ ディレクトリ（任意）

Claudeが必要に応じて読み込む追加ドキュメントを格納する。

用途：
- APIガイドやリファレンス
- 詳細な使用例やパターン集
- ドメイン固有の知識ベース

段階的開示（progressive disclosure）の仕組み：
- **第1レベル（YAMLフロントマター）**: 常にClaudeのシステムプロンプトに読み込まれる。スキルをいつ使うべきかの判断に必要な最小限の情報。
- **第2レベル（SKILL.md本文）**: Claudeがスキルが現在のタスクに関連すると判断したときに読み込まれる。完全なインストラクションとガイダンス。
- **第3レベル（リンクされたファイル）**: スキルディレクトリ内の追加ファイル。Claudeが必要に応じて参照する。

この仕組みにより、トークン使用量を最小化しつつ専門的な能力を維持できる。

## assets/ ディレクトリ（任意）

テンプレート、フォント、アイコンなど、出力生成に使用するリソースを格納する。

用途：
- レポートテンプレート
- ブランドアセット（ロゴ、フォント）
- 出力フォーマットの雛形

## フォルダ命名ルール

| ルール | 良い例 | 悪い例 |
|--------|--------|--------|
| kebab-case | `notion-project-setup` | `notionProjectSetup` |
| スペースなし | `my-cool-skill` | `My Cool Skill` |
| アンダースコアなし | `data-processor` | `data_processor` |
| 小文字のみ | `sprint-planner` | `SprintPlanner` |

## README.mdについて

- スキルフォルダ内には `README.md` を含めない
- すべてのドキュメントは `SKILL.md` または `references/` に配置する
- GitHubで配布する場合は、リポジトリのルートレベル（スキルフォルダの外）にREADMEを配置する

## コンポーザビリティとポータビリティ

- Claudeは複数のスキルを同時に読み込める。スキルは他のスキルと併用されることを前提に設計する
- スキルはClaude.ai、Claude Code、APIで同一の動作をする
- 外部要件がある場合は `compatibility` フィールドに明記する

## コンテキストサイズへの配慮

- SKILL.mdは5,000語以下に抑える
- 詳細なドキュメントは`references/`に分割する
- 20〜50以上のスキルを同時に有効にしている場合は、選択的な有効化を検討する
