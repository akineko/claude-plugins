# スキル設計パターン リファレンス

実績のあるスキル設計パターンをまとめたリファレンスです。ユーザーのユースケースに最適なパターンを選択してください。

## パターンの選び方

ユーザーの課題に合わせて2つのアプローチがある：

- 課題起点：「プロジェクトのセットアップをしたい」→ スキルが適切なツール呼び出しをオーケストレーション
- ツール起点：「Notion MCPを接続した」→ スキルが最適なワークフローとベストプラクティスを提供

## パターン1：順序実行型

決まった順番で複数ステップを実行するワークフローに使用する。

適用場面：
- オンボーディングフロー
- セットアップ手順
- 段階的なデータ処理

構造の例：
```markdown
## 手順

### ステップ1：アカウント作成
MCPツール `create_customer` を呼び出す。
パラメータ：名前、メール、会社名
成功条件：customer_idが返される

### ステップ2：支払い設定
MCPツール `setup_payment_method` を呼び出す。
前提条件：ステップ1のcustomer_idが取得済み
成功条件：支払い方法が検証される

### ステップ3：サブスクリプション作成
MCPツール `create_subscription` を呼び出す。
パラメータ：plan_id, customer_id（ステップ1から）
成功条件：サブスクリプションがアクティブになる
```

設計のポイント：
- ステップ間の依存関係を明示する
- 各ステップの成功条件を定義する
- 失敗時のロールバック手順を含める

## パターン2：反復改善型

品質チェックと修正を繰り返して出力の品質を上げるワークフローに使用する。

適用場面：
- レポート・ドキュメント生成
- コードレビューと修正
- デザインの磨き込み

構造の例：
```markdown
## 手順

### ステップ1：初期草案の作成
データを取得し、初期草案を生成する。

### ステップ2：品質チェック
バリデーションスクリプトを実行する：`scripts/check_report.py`
以下の基準でチェックする：
- 必須セクションがすべて含まれているか
- フォーマットが統一されているか
- データの整合性に問題がないか

### ステップ3：修正ループ
品質チェックで検出された問題を修正する。
修正後、ステップ2を再度実行する。
品質基準をすべて満たしたらステップ4に進む。
最大3回まで繰り返す。

### ステップ4：最終化
最終フォーマットを適用し、成果物を出力する。
```

設計のポイント：
- 明確な品質基準を定義する
- バリデーションはスクリプトで自動化することを推奨（コードは確定的、言語解釈はブレる）
- 繰り返しの上限回数を設定する（無限ループ防止）
- いつ「十分」とするかの判定基準を含める

## パターン3：コンテキスト判定型

状況に応じて異なる処理を選択するワークフローに使用する。

適用場面：
- ファイル種別による処理分岐
- ユーザーの入力内容に応じた対応
- 環境や条件による切り替え

構造の例：
```markdown
## 手順

### ステップ1：入力の分析
ファイルの種類とサイズを確認する。

### ステップ2：処理方法の決定

大容量ファイル（10MB超）の場合：
→ クラウドストレージMCPを使用

共同編集ドキュメントの場合：
→ Notion/Docs MCPを使用

コードファイルの場合：
→ GitHub MCPを使用

一時ファイルの場合：
→ ローカルストレージを使用

### ステップ3：処理の実行
決定した方法で処理を実行する。

### ステップ4：結果の説明
なぜその方法を選んだかをユーザーに説明する。
```

設計のポイント：
- 判定基準を明確に定義する
- フォールバックオプションを用意する
- 判断の透明性を確保する（なぜその方法を選んだか説明する）

## パターン4：ドメイン知識型

専門知識をスキルに埋め込み、ツールアクセス以上の価値を提供するワークフローに使用する。

適用場面：
- コンプライアンスチェック
- 専門的なレビュー
- ベストプラクティスの適用

構造の例：
```markdown
## 手順

### ステップ1：情報収集
MCPツールで対象データを取得する。

### ステップ2：専門ルールの適用
以下のルールに基づいてチェックする：
- [ドメイン固有のルール1]
- [ドメイン固有のルール2]
- [ドメイン固有のルール3]
チェック結果を記録する。

### ステップ3：判定と実行
チェックに合格した場合：
→ 処理を実行する

チェックに不合格の場合：
→ レビューのためにフラグを立てる
→ 不合格理由を記録する

### ステップ4：監査証跡
すべてのチェック結果と判定を記録する。
```

設計のポイント：
- ドメイン知識を具体的なルールとして記述する
- 判定前の検証を必須にする
- 監査証跡（なぜその判断をしたか）を残す

## パターン5：マルチMCP連携型

複数のサービスをまたぐワークフローに使用する。

適用場面：
- デザインから開発へのハンドオフ
- クロスプラットフォームの同期
- 複数ツールを使った報告

構造の例：
```markdown
## 手順

### フェーズ1：データ取得（サービスA MCP）
1. サービスAからデータをエクスポート
2. 必要な情報を抽出
検証：データが正しく取得されたか確認

### フェーズ2：データ保存（サービスB MCP）
1. サービスBにフォルダを作成
2. データをアップロード
3. 共有リンクを生成
検証：アップロードが完了したか確認

### フェーズ3：タスク作成（サービスC MCP）
1. 開発タスクを作成
2. データリンクをタスクに添付
3. 担当者をアサイン
検証：タスクが正しく作成されたか確認

### フェーズ4：通知（サービスD MCP）
1. 完了サマリーを投稿
2. リンクと参照を含める
```

設計のポイント：
- フェーズ間の明確な区切りを設ける
- フェーズ間のデータ受け渡しを明示する
- 各フェーズの完了を次のフェーズに進む前に検証する
- 一元的なエラーハンドリングを含める

## パターン選定のガイド

| ユースケース | 推奨パターン |
|---|---|
| 手順が決まっている定型作業 | 順序実行型 |
| 品質が重要な成果物作成 | 反復改善型 |
| 入力によって処理が変わる | コンテキスト判定型 |
| 専門知識が必要 | ドメイン知識型 |
| 複数サービスの連携 | マルチMCP連携型 |

複数のパターンを組み合わせることも有効。例えば、順序実行型の各ステップで反復改善型を適用するなど。

## バンドルリソースの活用

どのパターンでも、references/やscripts/を活用してSKILL.mdを簡潔に保つことが重要。

references/の活用例：
```markdown
クエリを書く前に `references/api-patterns.md` を参照：
- レート制限のガイダンス
- ページネーションパターン
- エラーコードとハンドリング
```

scripts/の活用例：
```markdown
`scripts/validate.py --input {filename}` を実行してデータ形式を検証する。
バリデーション失敗時のよくある問題：
- 必須フィールドの欠落（CSVに追加する）
- 日付形式の不一致（YYYY-MM-DDを使用）
```

SKILL.mdにすべてを書くのではなく、核心的な指示に集中し、詳細はリンク先に委ねる。
