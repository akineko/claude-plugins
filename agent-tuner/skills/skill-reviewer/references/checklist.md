# スキルレビュー チェックリスト

各項目の重大度:
- ❌ **要修正** — ガイドライン違反、または重大な問題がある
- ⚠️ **改善推奨** — 動作はするが改善の余地がある

---

## ファイル構造

### ❌ 要修正
- [ ] SKILL.mdが正確に `SKILL.md` と命名されている（大文字小文字を区別。`SKILL.MD`, `skill.md`, `Skill.md` などは不可）
- [ ] スキルフォルダ内に `README.md` が存在しない
- [ ] フォルダ名が「claude」や「anthropic」で始まっていない

### ⚠️ 改善推奨
- [ ] フォルダ名がケバブケースである（スペース、アンダースコア、大文字は非推奨）
  - 良い例: `notion-project-setup`
  - 悪い例: `Notion Project Setup`, `notion_project_setup`, `NotionProjectSetup`
- [ ] ディレクトリ構成が推奨パターンに沿っている:
  ```
  your-skill-name/
  ├── SKILL.md          # 必須
  ├── scripts/          # オプション
  ├── references/       # オプション
  └── assets/           # オプション
  ```

## YAMLフロントマター

### ❌ 要修正
- [ ] フロントマターが `---` のみの行で上下を区切られている
- [ ] `name`: 存在し、ケバブケースで、スペースや大文字がない
- [ ] `name`: 64文字以下
- [ ] `description`: 存在し、空でない
- [ ] フロントマター内にXMLの山括弧（`<` `>`）がない（セキュリティ上の制約: フロントマターはClaudeのシステムプロンプトに表示される）
- [ ] インジェクション攻撃に利用されうるような内容がフロントマターに含まれていない
- [ ] スキル名に「claude」や「anthropic」が含まれていない（予約語）
- [ ] YAMLの構文エラーがない（閉じ忘れのクォート、不正なインデントなど）

### ⚠️ 改善推奨
- [ ] `name` がフォルダ名と一致している
- [ ] `description` が1024文字以内である

### ⚠️ オプションフィールド（存在する場合の正しい形式）
- [ ] `license`: 標準的な識別子（例：MIT、Apache-2.0）
- [ ] `compatibility`: 1〜500文字、環境要件を記載
- [ ] `metadata`: キーバリューペア（author、version、mcp-serverなど）
- [ ] `allowed-tools`: スペース区切りのツールパターン

### 許可されたトップレベルフィールド
name、description、license、allowed-tools、metadata、compatibility のみ。カスタムデータは `metadata:` の下に配置すること。

## 説明文（description）

### ⚠️ 改善推奨
- [ ] WHAT（何をするか）が含まれている
  - 良い例: `Figmaデザインファイルを分析し、開発者向けハンドオフドキュメントを生成する`
  - 悪い例: `プロジェクトを手伝う`
- [ ] WHEN（いつ使うか）のトリガー条件が含まれている
  - 良い例: `ユーザーが.figファイルをアップロードしたとき、「デザインスペック」「コンポーネントドキュメント」を要求したときに使用`
  - 悪い例: トリガー条件の記述なし
- [ ] ユーザーが実際に言いそうな自然なトリガーフレーズを2〜3個含む
- [ ] 該当する場合、関連するファイル形式に言及している
- [ ] 主要な機能を記載している
- [ ] オーバートリガーを避けられる程度に具体的である
- [ ] 他のスキルとスコープが重なりうる場合、ネガティブトリガーを含む
- [ ] 他のスキルとの競合が起きそうなトリガー条件になっていない
- [ ] Claudeの組み込み機能とスコープが重複していない

### descriptionの構造テンプレート
```
[アクション動詞] [具体的なドメイン] [出力タイプ]。[トリガーフレーズ]の時に使用。
```

#### 良い例
- 「Linearプロジェクトワークフローを管理。スプリント計画、タスク作成、ステータス追跡を含む。ユーザーが「スプリント」「Linearタスク」「プロジェクト計画」に言及した時、「チケットを作成して」と依頼した時に使用。」
- 「PayFlowのエンドツーエンド顧客オンボーディングワークフロー。アカウント作成、支払い設定、サブスクリプション管理を担当。ユーザーが「新規顧客をオンボード」「サブスクリプションを設定」「PayFlowアカウントを作成」と言った時に使用。」

#### 悪い例（避けるべき）
- 曖昧：「プロジェクトを手伝います」→ トリガーなし、汎用的すぎる
- トリガーなし：「高度なマルチページドキュメントシステムを作成します」→ いつ使うか不明
- 技術的すぎる：「階層関係を持つProjectエンティティモデルを実装します」→ ユーザートリガーなし

## 指示本文（フロントマター以降）

### ❌ 要修正
- [ ] 指示が一切ない（フロントマターだけでSKILL.md本文がない）

### ⚠️ 構造
- [ ] 明確なMarkdown見出しで整理されている
- [ ] 連続するワークフローには番号付きステップを使用
- [ ] 重要な指示がファイルの上部に配置されている（埋もれていない）
- [ ] 最初から最後まで論理的な流れがある

### ⚠️ 明確さ
- [ ] 指示が具体的かつ実行可能である
- [ ] ツール/MCPの呼び出しがパラメータ名とともに明示されている
- [ ] 各ステップの期待される出力が記述されている
- [ ] 曖昧な表現がない
- [ ] Claudeが既に知っている一般知識の不要な解説がない（例：PDFとは何か、ライブラリのインストール方法など）
- [ ] 指示の具体性がタスクのリスクに見合っている（安全なタスクに過剰な手順指定がない、危険なタスクに十分なガードレールがある）
- [ ] 冗長すぎない（同じ内容が繰り返されていない）

#### 避けるべきアンチパターン
- 「データを適切に検証する」→ 具体的なチェック項目を指定すべき
- 「適切に処理する」→ ケースごとの具体的なアクションをリスト化すべき
- 「すべてが動作することを確認する」→ 具体的な成功基準を定義すべき
- すべてのコンテンツをインライン化 → references/を使った段階的開示を活用すべき

### ⚠️ エラーハンドリング
- [ ] よくある失敗パターンに対処している
- [ ] トラブルシューティングセクションが存在する
- [ ] 必要に応じてロールバック/リトライの手順がある

### ⚠️ 使用例
- [ ] 具体的な使用例が少なくとも1つ含まれている
- [ ] 例にトリガーフレーズ、実行ステップ、期待結果が示されている

### ⚠️ Progressive Disclosure

ガイドラインでは3段階のプログレッシブディスクロージャーが推奨されている:
- 第1レベル（YAMLフロントマター）: 常にシステムプロンプトに読み込まれる
- 第2レベル（SKILL.md本文）: スキルが関連する場合に読み込まれる
- 第3レベル（リンクされたファイル）: 必要に応じてClaudeが参照する

- [ ] SKILL.mdが約5,000語以内である（超える場合はreferences/への分離を推奨）
- [ ] 詳細な参考資料が `references/` に分離されインラインに埋め込まれていない
- [ ] SKILL.md本文が `references/` の内容と重複していない
- [ ] 参考ファイルへのリンクに「いつ読むべきか」の説明がある

### ⚠️ コンポーザビリティ

- [ ] 他のスキルと同時に動作する前提で設計されている（排他的な想定がない）
- [ ] 明確なスコープ境界が定義されている

### ⚠️ セキュリティ

- [ ] 外部リソースへのアクセスが適切に制限されている

### ⚠️ MCP拡張スキル向け（該当する場合）

- [ ] 使用するMCPツール名が正確に記載されている（大文字小文字を区別）
- [ ] MCP障害時の対処への言及がある
- [ ] MCPが利用できない場合の動作への言及がある（該当する場合）

## 補足ファイル

### ⚠️ 改善推奨
- [ ] `references/` のファイルが「いつ読むべきか」の指示とともにリンクされている
- [ ] `assets/` のファイルが指示の中で実際に使用されている
- [ ] SKILL.mdから参照されていない孤立ファイルがない

### スクリプト（scripts/ディレクトリがある場合）

#### ⚠️ 改善推奨
- [ ] 各スクリプトの呼び出し方法がSKILL.md内に具体的に記載されているか
  - コマンド、引数、オプションが明示されているか
  - 良い例: `python scripts/validate.py --input {filename}`
  - 悪い例: 「バリデーションスクリプトを実行する」
- [ ] 各スクリプトの期待される出力が説明されているか
- [ ] スクリプト失敗時の対処法が記載されているか
- [ ] 決定論的な処理（バリデーション、フォーマットチェックなど）にスクリプトが活用されているか
  - ガイドラインの原則:「コードは決定論的だが、言語の解釈はそうではない」
  - バリデーションを言語指示だけで行っている場合、スクリプト化を推奨する
- [ ] スクリプトにハードコードされた認証情報やAPIキーがない
