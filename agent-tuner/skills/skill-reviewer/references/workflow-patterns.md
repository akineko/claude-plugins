# ワークフローパターン別チェック項目

対象スキルが使用しているワークフローパターンを特定し、そのパターン固有のベストプラクティスが守られているかを確認する。

各項目の重大度:
- ❌ **要修正** — ガイドライン違反、または重大な問題がある
- ⚠️ **改善推奨** — 動作はするが改善の余地がある

---

## パターンの特定方法

スキルの指示内容から、以下のどのパターンに該当するか判断する。複数のパターンを組み合わせているスキルもある。

---

## パターン1: 逐次ワークフロー（Sequential Workflow Orchestration）

**特徴**: 複数ステップを特定の順序で実行する

### チェック項目

#### ❌ 要修正
- [ ] ステップの順序が明確に定義されているか
- [ ] ステップ間の依存関係が明示されているか（例: ステップ3はステップ1のcustomer_idを使用）

#### ⚠️ 改善推奨
- [ ] 各ステップにバリデーションがあるか
- [ ] 失敗時のロールバック手順が記載されているか（途中で失敗した場合にどうするか）

### 良い例
```
# ステップ1: アカウント作成
MCPツール `create_customer` を呼び出す
パラメータ: name, email, company
成功時: customer_idを保持

# ステップ2: 支払い設定
MCPツール `setup_payment_method` を呼び出す
前提: ステップ1のcustomer_idが必要
失敗時: ステップ1で作成したアカウントの状態を「保留」に変更
```

---

## パターン2: マルチMCP連携（Multi-MCP Coordination）

**特徴**: 複数の異なるMCPサーバーをまたいでワークフローを実行する

### チェック項目

#### ❌ 要修正
- [ ] フェーズ（利用するMCPサーバーごと）が明確に分離されているか
- [ ] フェーズ間のデータ受け渡し方法が明示されているか

#### ⚠️ 改善推奨
- [ ] 次フェーズに移行する前のバリデーションがあるか
- [ ] エラーハンドリングが一元化されているか（各フェーズのエラーを統合的に扱う仕組み）

### 良い例
```
# フェーズ1: デザインエクスポート（Figma MCP）
# フェーズ2: アセット保存（Drive MCP）
# フェーズ3: タスク作成（Linear MCP）
# フェーズ4: 通知（Slack MCP）
```

---

## パターン3: 反復改善（Iterative Refinement）

**特徴**: 出力品質をイテレーションで改善する

### チェック項目

#### ❌ 要修正
- [ ] イテレーションの停止条件があるか（無限ループにならないための基準）

#### ⚠️ 改善推奨
- [ ] 品質基準が明示的に定義されているか（何をもって「良い」とするか）
- [ ] バリデーションスクリプトやチェック手段が提供されているか
- [ ] 初回ドラフト → チェック → 改善 → 再チェック の流れが明確か

### 良い例
```
# 品質チェック
scripts/check_report.py を実行し、以下を確認:
- 必須セクションの存在
- フォーマットの一貫性
- データバリデーションエラー

# 改善ループ
各指摘事項に対応 → 再バリデーション → 閾値を満たすまで繰り返し
最大3回のイテレーションで停止
```

## パターン4: コンテキスト依存ツール選択（Context-Aware Tool Selection）

**特徴**: 状況に応じて異なるツールやアプローチを選択する

### チェック項目

#### ❌ 要修正
- [ ] 判断基準（ディシジョンツリー）が明確か

#### ⚠️ 改善推奨
- [ ] 各選択肢の条件が具体的か（例: 10MB超 → クラウドストレージ）
- [ ] フォールバックオプションが用意されているか（第一選択が使えない場合）
- [ ] 選択理由がユーザーに説明される仕組みがあるか（透明性）

### 良い例
```
# 判断基準
1. ファイルの種類とサイズを確認
2. 保存先を決定:
   - 大きいファイル（10MB超）: クラウドストレージMCP
   - 共同編集ドキュメント: Notion/Docs MCP
   - コードファイル: GitHub MCP
   - 一時ファイル: ローカルストレージ
3. フォールバック: 第一選択が失敗した場合はローカル保存
```

---

## パターン5: ドメイン特化（Domain-Specific Intelligence）

**特徴**: ツールアクセスを超えた専門知識を付加する

### チェック項目

#### ⚠️ 改善推奨
- [ ] ドメイン固有のルールやロジックが具体的に記述されているか
- [ ] アクション実行前のチェック（コンプライアンス、バリデーション等）があるか
- [ ] 判断の監査証跡（ログ、レポート）を残す仕組みがあるか
- [ ] ドメインルールが明確で曖昧さがないか

### 良い例
```
# コンプライアンスチェック（処理前）
1. 取引詳細をMCPで取得
2. コンプライアンスルールを適用:
   - 制裁リストの照合
   - 管轄区域の許可確認
   - リスクレベルの評価
3. コンプライアンス判定を記録

# 処理
コンプライアンス合格 → 決済処理を実行
コンプライアンス不合格 → レビューにフラグ
```
