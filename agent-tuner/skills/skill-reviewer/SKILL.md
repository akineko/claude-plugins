# スキルレビュアー

Claudeスキルをベストプラクティスに基づいてレビューし、具体的な改善レポートを生成するスキルです。

## 重要な原則

- **建設的であること**: 問題を「失敗」ではなく「改善の機会」として伝える。ユーザーがより良いスキルを完成させることが目標。
- **インパクト順に提示**: 最も効果の大きい改善から先に伝える（通常はdescriptionの品質とトリガーの問題）。
- **見せる、言うだけでない**: 書き直しを提案するときは、アドバイスだけでなく実際の修正テキストを提供する。
- **意図を尊重する**: スキルの目的を再設計しない。その目的をより良く伝え、実行する方法を改善する。
- **重大度を正直に**: ❌（動作しない）、⚠️（品質低下）、✅（良好）を明確に区別する。

## レビュー手順

以下のステップを順番に実行すること。ステップを飛ばさないこと。

### ステップ1: スキルの取得

レビュー対象を特定する：

- ユーザーがパスを指定した場合 → `Read` ツールで読み込む
- ユーザーがチャットに内容を貼り付けた場合 → その内容をそのまま使用する
- パスが不明な場合 → `Glob` ツールで `**/SKILL.md` を検索する

SKILL.mdの全内容を確認してから次に進むこと。スキルフォルダにscripts/、references/、assets/が含まれる場合、その存在をステップ6のために記録しておく。

### ステップ2: ファイル構造チェック

`references/checklist.md` の「ファイル構造」セクションに従い、各項目を判定する。❌の項目には簡潔な理由を付記すること。

### ステップ3: YAMLフロントマターチェック

`references/checklist.md` の「YAMLフロントマター」セクションに従い、各項目を判定する。❌の項目には簡潔な理由を付記すること。

### ステップ4: 説明文（description）の品質評価

descriptionはスキルの中で最も重要な部分である。Claudeがスキルを読み込むかどうかを決定する。

`references/checklist.md` の「説明文（description）」セクションに基づいて評価する。

### ステップ5: 指示内容の品質レビュー

フロントマター以降のSKILL.md本文を評価する。`references/checklist.md` の「指示本文」セクションの各サブセクションに基づいて判定する。

### ステップ6: 補足ファイルのレビュー（存在する場合）

スキルに `scripts/`、`references/`、`assets/` が含まれる場合、`references/checklist.md` の「補足ファイル」セクション（スクリプトの項目を含む）に基づいて検証する。

### ステップ7: スキルカテゴリとパターンの評価

スキルがどのカテゴリ・パターンに該当するかを判定し、カテゴリ固有のベストプラクティスが守られているかを確認する。

- **カテゴリ判定**: `references/category-checks.md` を参照し、3カテゴリ（ドキュメント＆アセット生成 / ワークフロー自動化 / MCP連携強化）のどれに該当するかを判定。該当カテゴリの追加チェックを実施する。
- **パターン判定**: `references/workflow-patterns.md` を参照し、5パターン（逐次ワークフロー / マルチMCP連携 / 反復改善 / コンテキスト依存ツール選択 / ドメイン特化）のどれに該当するかを判定。該当パターンのチェック項目を確認する。

複数のカテゴリやパターンにまたがるスキルもある。その場合は該当する全てのチェックを行う。

### ステップ8: レビューレポートの生成

以下の骨格と表示ルールに従いレポートを生成する。

#### 評価記号

| 記号 | 意味 |
|------|------|
| ✅ | 問題なし — チェック項目を満たしている |
| ⚠️ | 改善推奨 — 動作するが改善の余地がある |
| ❌ | 要修正 — ガイドライン違反、または動作しない |

記号は各参照ファイル（checklist.md, category-checks.md, workflow-patterns.md）の項目に付与された重大度に従う。

#### レポート骨格

レポートは以下のセクション構成とする。`→` はチェック項目の参照元を示す。

1. **総合評価**
2. **ファイル構造** → checklist.md「ファイル構造」
3. **YAMLフロントマター** → checklist.md「YAMLフロントマター」
4. **説明文の品質** → checklist.md「説明文（description）」
5. **指示の品質** → checklist.md「指示本文」の全サブセクション
6. **補足ファイル** → checklist.md「補足ファイル」
7. **スキルカテゴリ・パターン** → category-checks.md + workflow-patterns.md
8. **改善提案（優先度順、最大3件）**

#### 総合評価の判定基準

ステップ2〜7の全❌/⚠️を合算し、上から順に最初に該当するものを適用:
- **D**: ❌が3件以上、またはdescriptionに❌がある
- **C**: ❌が1〜2件
- **B**: ❌が0件、⚠️が3件以上
- **A**: ❌が0件、⚠️が2件以下

総合評価セクションには以下を含める:
- 1〜2文の全体的な評価コメント
- **強み**: このスキルの優れた点を1〜2項目
- **要改善**: 主な問題を重大度マーク付きで1〜3項目

#### セクション表示ルール

各セクション（2〜7）は、対応するチェック項目をサブセクション単位で1つずつ評価した上で、以下のルールで表示する:

- **全項目✅** → セクション見出し + `問題なし` の1行
- **⚠️/❌あり** → 問題のある項目のみ列挙（✅の項目は省略してよい）
- 補足ファイルセクションのみ、scripts/・references/・assets/のいずれも存在しない場合はセクション自体を省略してよい

項目の表示形式: `- **[チェック項目の要約]**: [✅/⚠️/❌] — [問題がある場合のみ具体的な指摘]`

#### 改善提案セクション

❌の項目は全て提示する。⚠️の項目はインパクト上位3件まで。
提示順:
1. 説明文の問題（スキルが読み込まれるかを左右するため最優先）
2. ❌の項目（動作しない問題）
3. ⚠️の項目（品質低下）

各項目には以下を含む:
- なぜ重要か（1文）
- 具体的な修正例（「現状 → 修正案」形式）

複数の修正がフロントマター内など同一箇所に集中する場合は、個別ではなく統合した修正例を提示する。

## 使用例

### 例1：パス指定でクイックレビュー
ユーザーの発言：「`.claude/skills/my-skill/SKILL.md` をレビューして」
アクション：
1. `Read` ツールでSKILL.mdを読み込む
2. フォルダ内の他のファイルを `Glob` で確認
3. ステップ2〜7のレビューを実施
4. 深刻度レベル別に分類されたレポートを提示
5. 改善版の生成を提案

### 例2：説明文のみのレビュー
ユーザーの発言：「このスキルの説明文は良い？」（テキストを貼り付け）
アクション：
1. 説明文の品質基準に照らして評価（ステップ4）
2. 改善版を提案

### 例3：新規スキル開発後のチェック
ユーザーの発言：「新しく作ったスキルが正しいか確認して」
アクション：
1. スキルのパスを確認、または `Glob` で検索
2. 完全なレビュー（ステップ2〜7）を実行
3. 問題があれば修正版を提示

## トラブルシューティング

### ユーザーがSKILL.mdの内容のみを提供した（ファイルではない）
原因：チャットに直接コンテンツを貼り付け
解決策：貼り付けられた内容をそのままレビュー対象として使用する。ファイル構造チェック（ステップ2）はスキップし、その旨をレポートに記載する。

### スキルがMCPを使用しているがレビュー環境でMCPサーバーが接続されていない
原因：レビュー時にMCPサーバーが利用不可
解決策：MCPツール参照を構文的にレビューする（ツール名の明示、パラメータの記載、エラーハンドリングの有無）。ランタイムテストには実際のMCP接続が必要であることをレポートに注記する。

### スキルのパスが不明
原因：ユーザーがスキル名のみを伝え、パスを指定しなかった
解決策：`Glob` ツールで `**/<スキル名>/SKILL.md` または `**/SKILL.md` を検索する。複数見つかった場合はユーザーに選択を求める。

### スキルにスクリプトが含まれるが実行環境がない
原因：スクリプトの言語やランタイムがレビュー環境にインストールされていない
解決策：スクリプトの内容を `Read` ツールで読み込み、コードレビューとして評価する。実行テストが必要な場合はその旨をレポートに注記する。
